REPLY 2 (by nckw @ 2024-10-14T15:57:03.303Z):
Hi Fabio,
The ChannelCompatibilityCheck really only works with defining one signal strength per channel vs one overall signal strength, since the method itself produces the parameters. If you wanted to check the compatibility of another parameter (eg mass in each category), you could instead define a signal model for it and calculate the deltaNLL between the signal and multi version using the MultiDimFit.
All the best,
Nick

REPLY 4 (by amarini @ 2024-10-15T08:27:18.422Z):
Hi Fabio,
the question is if your POI acts as a strength modifier or not, which is unclear to us.
In general what you wrote seems correct. It should be better to encompass both model (w/ and w/o splitting) in the same workspace definition. Then, you can run the fit a) with the single POI and the fit b) with N pois (notice that the two fits are nested) and then you can use the -2\Delta\log\mathcal{L}  as a \chi^2 distributed variable (Wilks’ th) with the right number of d.o.f.
For example (if your poi act like this on your likelihood) (with some abuse of notation) (A):
\mathcal{L} = \prod \mathcal{P}( \mu S + B \dots) \dots
but not like these (B):
\mathcal{L} = \prod \mathcal{P}( \mu^2 S + B \dots or \mathcal{L} \prod \mathcal{P}( \mu (1-\mu)S \dots) \dots
or generally (like a mass or width, C):
\mathcal{L} = \prod \mathcal{P}( S(\mu) + B \dots)
If the answer is yes (case A), probably channel compatibility already works as it takes the first poi. If the answer is no (cases B or C), even if it may work out of the box, the outcome won’t likely have a useful meaning and you need to follow a more complex procedure. (as explained by Nick).
If the POIs are multiplicative (act as a mutiplicative term to some process/cat, i.e. case (B) but not a mass measurement like case (C)), you could also use MultiSignalModel to generate the pairing on the fly.
text2workspace.py -P HiggsAnalysis.CombinedLimit.PhysicsModel:multiSignalModel --PO 'map=POI_catXXX_yearYY/signal:r[0.5,0,2]*rXY[0.5,0,2]*(1-r)*(1-rXY)*4' ...
combine -M MultiDimFit --redefineSignalPOIs rXY,... --freezeParameters r --setParameters r=0.5

(IIRC, you have to define the parameters all consistently in roofit or maybe only the first time. I don’t recall the extend we can use regex in matching groups in the definitions above.)
-Andrea

REPLY 5 (by ineuteli @ 2024-10-15T08:50:36.787Z):
Hi Fabio, all
If this is helpful at all: combine -M ChannelCompatibilitiyCheck has an inbuilt -g/--group option that allows you to group together several bins (categories × years). If I remember correctly, if you have bins $CAT_$ERA you could try something like the following:
combine -M ChannelCompatibilitiyCheck ... -g cat1_2016 -g cat2_2016 -g cat1_2017 -g cat2_2017 ...

However, it does not allow regular expression, so it might not work depending on the naming of your bins. If there’s some overlap in the naming, you have to get creative. (Play with the order of the -g options might help.)
Note that you can also set the ranges of the signal strength for individual groups to help the fit converge if these groups are less sensitive: -g cat1_2016=0.5,1.5, etc.
I am also not 100% sure if it allows parameter called anything other than the default r, but quickly looking at the source code [1], I think it will take whatever is defined as the first POI in the model via mc_s->GetParametersOfInterest()->first()? So you could probably redefine the POI with the command line option -P mypoi/--parameters mypoi?
Hope that helps,
Izaak
[1] https://github.com/cms-analysis/HiggsAnalysis-CombinedLimit/blob/main/src/ChannelCompatibilityCheck.cc

REPLY 7 (by fiemmi @ 2024-10-27T14:39:19.247Z):
Dear all,
thanks a lot for your replies and please accept my apologies for providing feedback this late. At the beginning I kept the question general on purpose, thinking that more people could find this thread in the future and profit from the discussion. I see, though, that the answer depends on the kind of POI in question. So, let me clarify that my POI is the Higgs mass, MH.
What I ended up doing, after discussing with @jlangfor as well (thanks a lot!), is the following:

I manipulated my input workspace(s) in order to effectively have one mass POI for each category and year. I ended up with a set of 20 POIs (5 categories times 4 years) of the form: MH_Cat0_2016_preVFP, MH_Cat1_2016_preVFP, [...], MH_Cat4_2016_preVFP, [...], MH_Cat4_2018.
In order to extract the value of the mass in each category and year, properly accounting for correlations, I fitted the aforementioned workspaces with -M MultiDimFit --algo singles.
In order to compute the degree of compatibility between the mass values in each category/year and the full Run2 combination, I fitted the aforementioned workspaces with -M MultiDimFit --algo fixed, where I fixed the values of the masses to the full Run2 combination value. I finally plugged the deltaNLL value obtained with this routine into ROOT.TMath.Prob(2*deltaNLL,ndof-1) and obtained the p-value for the compatibility.

I hope this makes sense to you. Please let me know if you have any concerns or comments.
Thanks, and all the best!
Fabio.
