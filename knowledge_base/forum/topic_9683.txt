REPLY 2 (by agilbert @ 2022-04-26T07:34:33.774Z):
Hi Federica,
So RooParametricHist is a RooFit pdf, and I think in principle it could be fed in to a RooMultiPdf alongside an alternative RooParametricHist. I’m not aware that anyone ever used RooMultiPdf or discrete profiling with binned distributions however, and maybe there are some technical issues that would pop up along the way.
If I understand correctly you would have:
CR1 - RooParametricHist A, with free parameters, one per bin (?) a_i
CR2 - RooParametricHist B, with free parameters b_i
SR - RooMultiPdf { RooParametricHist SR_A (functions of a_i), RooParametricHist SR_B (functions of b_i) }
And the idea is in the discrete profiling you will let the fit find which of the two extrapolations (CR1 → SR) or (CR2 → SR) works best?
It’s an interesting approach - If you go ahead with it it might be useful to start with a simplified test case, to see if it works as expected. I’d be interested to know the outcome.

REPLY 4 (by agilbert @ 2022-04-27T13:11:30.205Z):
Hi Federica,
Ok yes, so indeed this is an issue. I don’t think the RooExtendPdf approach will work very easily, would require modifying quite a few parts of combine. Other option is to leave the overall normalisation floating with a free parameter, but I guess this is also not what you want to do. I can see one way to write a normalisation RooFormulaVar that uses the discrete category index value to propagate the correction normalisation term:
import ROOT
w = ROOT.RooWorkspace()
w.factory('testCat[chnA=0,chnB=1]')
w.factory('normA[100]')
w.factory('normB[120]')
# Exploit that RooFormulaVar will evaluate a RooCategory as an integer
w.factory('expr::normterm("(1-@0)*@1 + @0*@2",testCat, normA, normB)')
w.cat('testCat').setIndex(0)
w.function('normterm').Print()
w.cat('testCat').setIndex(1)
w.function('normterm').Print()

gives:
RooFormulaVar::normterm[ actualVars=(testCat,normA,normB) formula="(1-@0)*@1+@0*@2" ] = 100
RooFormulaVar::normterm[ actualVars=(testCat,normA,normB) formula="(1-@0)*@1+@0*@2" ] = 120

It’s a bit ugly, but it might work. One caveat: if the number of degrees of freedom are different in the two norm terms, this won’t be taken into account in the NLL penalty term that is added when using the envelope method (the counting is only of parameters in the pdf, not the norm term). So this may or may not be a problem.
cheers,
Andrew
