REPLY 2 (by anigamov @ 2025-06-19T03:10:34.436Z):
Dear Lokesh,
To assign the lnN uncertainty for any process, signal or background, please follow the documentation here for example.
If you don’t want to modify your signal prediction with freely floating parameter, you can indeed freeze the signal strength  with --setParameters r=1 --freezeParameters r if you are using the default physics model. If there is another parameter that you are interested in it would be a good idea to make sure that Combine knows it with --redefineSignalPOIs <POI_name>, more details here.

REPLY 4 (by anigamov @ 2025-06-20T15:09:09.670Z):
It depends on what you are trying to achieve exactly. But I don’t see any reason to implement a constrain term on the normalization parameter itself if you just want to assign theory motivated systematics.
Let me try to explain a bit. When you assign the lnN systematics two things happen: the k^{\nu} term added to the primary likelihood component, to modify the normalization of a given process; then the auxiliary part of the likelihood gets an additional component (Gaussian in case of lnN) which depends on \nu. The freely floating normalization parameter does not change the behavior, but the effect is there. So I would say you are covered with the lnN’s, you can read more about this in the docs.
About the impacts on frozen parameters, to avoid having it plotted just add --exclude r to the impacts command. The impacts are estimated in parallel, and if you have it frozen during the initial fit and when the impacts from the other parameters are estimated, it does not affect anything at all.
Let me know if something is not clear,
Aliya
