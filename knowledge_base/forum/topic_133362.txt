REPLY 3 (by agilbert @ 2025-10-23T10:45:03.163Z):
Dear Kuan-Yu,
Let me try to add a few comments on the behaviour of the analytic minimisation here.

klin:

I found that the analytical solution tended to move quickly towards having autoMCstats NP shifted for the best-fit value when r is still at the initial value.

The way the analytic minimisation works is that when enabled, those nuisance parameter RooRealVars are set to frozen, and our CMSHistFuncErrorPropagator class will now set them to the best-fit value automatically whenever the model pdf is evaluated. Then, the minimizer object is created. When RooFit sets up the Minuit2 minimizer, it does not pass frozen RooRealVars. From the start of the fit, the MC stat parameters are set to their best-fit, given the initial values of r and all other floating parameters. As the fit progresses, they are updated to a new analytic solution any time another parameter value is changed by Minuit.

klin:

And then when RooFit float r, it is unable to use the Hessian matrix to go toward the global minimum because the latter requires to move the autoMCstats NP back.

I’m afraid I don’t fully understand this part. If you see error messages about the Hessian, perhaps being non-positive-definite, this suggests a problem in the shape of your (MC-stat-profiled) likelihood function. It seems what you are seeing is the presence of local minima when the analytic minimisation is enabled - the MC stat parameters are moved to some region of parameter space at the beginning, that fit then converges to a different minimum compared with Minuit stepping all parameters from their initial value. One would have to look carefully into the model and the effect of other parameters to understand why this local minimum occurs.

klin:

I am thinking if we can let autoMCstats NP float in RooFit and just use the analytical solution to enhance the minimisation speed?

I guess you mean leave floating in the Minuit fit, but I don’t see how to use the analytic solution in that case - if we hand over a free parameter to Minuit then we have no more control over it until the fit ends. One could imagine a setup where we first: freeze MC stats parameters, disable their analytic minimisation and minimize wrt the other params. Then enable MC stat analytic minimisation and run the fit again. This might avoid your double minima (assuming that is the underlying problem), but it’s not guaranteed. This procedure will be a bit tricky to test with toys unfortunately (though a few modifications in the combine fitter code could probably do it).
Cheers,
Andrew

This is not exactly correct. The minimisation is still smooth, and for a well-defined model there is no issue with computing the Hessian. Updated as a function of what? When we compute the total uncertainty in a bin we account for the overall normalisation scaling of each process (which multiplies both the contents and error). Doing this correctly is rather difficult, and what we do now should be a reasonable approximation in most cases (though it was not studied in detail).

REPLY 6 (by klin @ 2025-12-05T15:37:42.002Z):
Hi @agilbert,
I think I have found the root cause. If I understand your words and RooFit correctly, when MIGRAD go around the parameter space in iteration the function “CMSHistErrorPropagator::eval()” would find the best-fit autoMCStats NP for the given floating parameters. Therefore the Hessian matrix estimated by RooFit effectively include the change of NLL due to autoMCStats NPs and the EDM is still equivalent to the full NLL with those NPs floating.
I now believe the problem is that there are too many bins in this workspace have a high chance of having 0 data in each toy dataset, and one of the two analytical solution in CMSHistErrorPropagator become invalid. Maybe this is too trivial for you but let me give the full derivation process.
Assume that data = N_\text{data}, nominal prediction = n with \delta n MC errors, which was varied by the NP \theta. And the NP has global observable \theta_0. Then the NLL is
-N_\text{data}\log(n+\theta\delta n)+n+\theta\delta n+(\theta-\theta_0)^2/2.
We find the extrema by solving:
-N_\text{data}\delta n/(n+\theta\delta n)+\delta n +\theta-\theta_0 = 0 (Eq 1)
\implies (\theta-\theta_0+\delta n)(\theta+n/\delta n) = N_\text{data} (Eq 2)
If N_\text{data}>0, we get the two analytical solutions found in Combine, but only the large one is the true extremum because the other can be proved to be less than n/\delta n, rendering the log operation undefined.
But funny thing happens when N_\text{data}=0. The first term in Equation 1 disappears, and there is only one solution: \theta_\text{true} = \theta_0-\delta n. But the analytical solution is based on Equation 2, which will compare the only true solution to the non-existing solution \theta_\text{false} = -n/\delta n. If \theta_\text{true} \geq \theta_\text{false}, which is usually the case when r\geq0, then the solution taken by Combine is correct; otherwise, it were wrong.
But the false solution has an interesting property, the shifted bin prediction would be exactly 0! So if the true solution is less than the false solution, this means that the shifted prediction, n+\theta\delta n, is less than 0, an unphysical prediction. But unlike the N_\text{data}>0 situation, there is no log to evaluate in the likelihood function. So this unphysical solution is mathematically well defined. And Combine effectively force the shifted prediction to be non-negative.
I believed the divergence between the default behaviour and the alternative with --X-rtd MINIMIZER_no_analytic is due to the unphysical solution allowed by the latter. So I tested by hypothesis by adding the following line:
if(fabs(bb_.dat[j])<1e-8) bb_.res[j] = bb_.gobs[j] - bb_.toterr[j];

after this line. And indeed I recover the low-bias pull plot and a symmetric r distribution, as what --X-rtd MINIMIZER_no_analytic had produced.
From this reasoning, the problem can only appear when n+\theta_\text{true}\delta n<0. This occurs frequently in this workspace not only because it has a lot bin with N_\text{data}=0, but also \theta_0 going negative in frequentist toys. When the analyser set asymptotic limit, \theta_0 would be fixed to 0. Then even if N_\text{data} is indeed 0, n+\theta_\text{true}\delta n = n-\delta^2 n. This identity can be < 0 if r is very negative such that n=rs+b<0 or if the average weights are large. So perhaps the exclusion limit based on \tilde{q}_\mu is unaffected. But I am still curious if allowing unphysical solution would allow the asymptotic approximation work better in low data analysis based on the symmetric r distribution I found in this workspace.
May I ask your recommendation on this situation? Should we keep Combine as it is and deem this bias/pull test from toys not useful because it has high chance to bump into the unphysical bound for the model? Or we should add an option in Combine to allow this unphysical solution to check if the systematic templates were well defined?

Hi Kuan-Yu,
Thanks for looking into this in detail.
The special case where the total number of expected events in a bin is <= 0 is always problematic. Indeed if one is just setting upper limits it should be less of a problem, otherwise we advise people to try and avoid this situation if possible, perhaps by rebinning distributions. Even if combine manages to work around this when Ndata = 0, I think there is no guarantee the subsequent results are really correct. That said, I think we may try and fix the behaviour of the analytic solution so it’s at least consistent with the non-analytic.
In any case, if the relative MC stat. uncertainty in the bin is quite large, such that toys regularly lead to the total yield < 0, it suggests the modelling with a single Gaussian is anyway not really a good approximation of the real likelihood, and it would be better to adjust the threshold in the datacard to treat these with a separate parameter per process. Did you try this already?
