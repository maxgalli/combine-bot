REPLY 2 (by clange @ 2025-06-27T10:23:56.602Z):
Hi Jaana,
I’ve coincidentally seen this message at CMSDAS this week and the reason was that the user had unknowingly set up an LCG release through their ~/.bashrc file (look also in ~/.profile and ~/.zshrc). You can also check this by looking at environment variables such as $PYTHON3PATH and $PYTHONPATH, which should only contain path values starting with /cvmfs/cms.cern.ch/ or your CMSSW directory. $PYTHONPATH is empty in my case. A Conda environment might also be the reason for this.
Cheers,
Clemens

REPLY 4 [ACCEPTED ANSWER] (by kcormier @ 2025-07-02T14:50:33.964Z):
Hi Jaana,
I’ve had people ask me about a similar-seeming (but not identical) issue in the past where it has turned out to be due to them having a different version of numpy installed locally. I wonder if this might be the problem you are seeing.
(From my notes some time ago) for a clean environment following the setup instructions on lxplus where everything was working, if I opened python and checked which numpy it was picking up, I’d get:
>>> import numpy
>>> numpy.__file__
'/cvmfs/cms.cern.ch/el9_amd64_gcc12/external/py3-numpy/1.24.3-ac08ea497df571aa0b37dc29bb7a2045/lib/python3.9/site-packages/numpy/__init__.py'
>>> numpy.__version__
'1.24.3'

Whereas someone who was getting numpy incompatibility errors with text2workspace (although, again a slightly different error) had numpy version 2.0.2 which was coming from their local installation with ~/.local/lib/python3.9/site-packages/numpy/__init__.py. Perhaps you have a similar local install which is getting picked up by default?
In their case they removed the local install  because there was no real reason for them to have it in the first place. After that things worked fine for them following the instructions.
Hope that helps,
Kyle

REPLY 6 (by ineuteli @ 2025-07-14T09:32:01.174Z):
Hi all,
Late to the party, but in case the following is useful to anyone, you can set following environmental variable to have python ignore local installations:
export PYTHONNOUSERSITE=True # do not add user site-packages in ~/.local to sys.path

see the official documentation on PYTHONNOUSERSITE and ENABLE_USER_SITE. In this way, you do not need to delete local installations.
I also ran into clashes between CMSSW and local python installations several times, which can manifest in many ways, even mysterious compilation issues… The CMSSW Core experts recommended me the method above.
Cheers,
Izaak
