REPLY 4 (by amarini @ 2022-10-07T09:05:46.950Z):
Hi,
IIRC, PostFitShapesFromWorkspace uses the hessian to derive the postfit uncertainties.
Combine usually use toys to compute the postfit shapes. There are ways to derive postfit from toys manually, i.e., by generating (GenerateOnly) fitting (MultiDimFit) toy-by-toy and later drawing the postfit shape and uncertainties.
A nice contribution you can implement is the automation of this procedure in steps of FitDiagnostics, similarly to what is done for other heavy steps like AsymptoticLimits or Hybrid.

REPLY 5 (by adewit @ 2022-10-07T09:20:35.475Z):
Hi Dennis,
I would support others’ suggestions for using PostFitShapesFromWorkspace. You’ll still need the fit results from FitDiagnostics since indeed PostFitShapesFromWorkspace needs to resample from the covariance matrix [*] to compute the uncertainty bands (FitDiagnostics does this too, by the way, but with PostFitShapesFromWorkspace you can manually parallellize the evaluation of the shapes for different categories, as suggested earlier in the thread).
One thing I want to point out is that you are using --ignoreCovWarning. I assume this means your covariance matrix was not positive-definite. The flag suppresses warnings related to that, but the warning is there for a reason: if the covariance matrix was not accurate, the uncertainty bands on the post-fit distributions will be incorrect. I believe in your analysis robustHesse was used in other places to avoid this issue, you should probably do the same here to ensure the covariance matrix is accurate.
Best,
Adinda
[*] If the --sampling option in PostFitShapesFromWorkspace is passed. It should be switched on by default, if my memory is correct.

REPLY 8 (by agilbert @ 2022-10-14T10:33:22.984Z):
Hi Peter,
Just to try and clarify a bit: the process of randomly sampling nuisance parameter sets from the covariance matrix to generate the unceratainty bands for the post-fit shapes is exactly the same for FitDiagnostics and PostFitShapesFromWorkspace. You can trace the same call to randomizePars for the former to here. Therefore the (multivariate) Gaussian approximation is assumed in both cases for sampling. The line of code you found in FitDiagnostics.cc is used for pre-fit uncertainties, and refers to the nuisance parameter pdf, not the pdf of the observables.
In both implementations, the generation of the sampled nuisance parameter values is not the bottleneck, it’s mostly re-evaluating the model each time and extracting the binned histograms. FitDiagnostics can be a little slower, mostly because there are other things it’s doing in addition (e.g. it does both b-only and s+b uncertainty bands by default, as well as computing some other post-fit information).
Hope this helps.
cheers,
Andrew
