QUESTION (by beozek @ 2026-05-20T21:30:07.820Z):
Hi all,
I am preparing the Combine/statistical review material for a TOP analysis and I am trying to run the recommended pre-fit Asimov closure checks from the TWiki.
The recommendation says to run:
combine -M FitDiagnostics -t -1 --expectSignal 0

and check that the fit gives r = 0, then run diffNuisances.py, and similarly run:
combine -M FitDiagnostics -t -1 --expectSignal 1

and check that the fit gives r = 1.
In my analysis, however, the POI is not the standard signal strength r. The POI is a template parameter called NoAC, which controls the charge-asymmetry template. The parameter r is not used as the physics POI and is frozen to 1.
For the nominal SM Asimov closure, I tried:
combine -M FitDiagnostics workspace_inclusive.root \

--redefineSignalPOIs NoAC \

--setParameters r=1,NoAC=0 \

--setParameterRanges NoAC=-20,20 \

--freezeParameters r \

-t -1 \

--robustFit 1 \

-n _closure_inclusive

The fit returns:
Best fit NoAC: 0.000299433

Here is the output log:

REPLY 2 (by gortona @ 2026-05-20T22:21:02.677Z):
Dear Beren,
I think what you are doing is correct. You should have a first test with --setParameters r=1,NoAC=0 and a second one with --setParameters r=1,NoAC=1. This assuming that NoAC=0 is your background-only situation and NoAC=1 is the SM expectation. Testing -1 as you suggest seems also a good idea.
Basically you want to make sure that you fit back whatever value of the poi you inject with setParameters by testing a few special cases.
expectSignal=X is basically the same as setting setParameters r=X, it is just a shorthand for the standard POI r. If you are using another POI you will need to use setParameters but it will still be fine.
About diffNuisances, you can specify a POI different from r with the -p option, such as -p NoAC
Best,
Giacomo

REPLY 4 (by gortona @ 2026-05-21T07:36:27.386Z):
Hi Beren,
Sorry now I understood your problem better. I read a bit too quick your previous message.
I would suggest to try the following options (in order):

Try to limit the range of NoAC between -5 and 5
use --robustFit (I saw you were already using it in previous commands and it seemed to improve a bit the -ve uncertainty), coupling it with --stepsize=0.001. You can also try to tune the --setRobustFitTolerance and --setCrossingTolerance options.
If these fail, try to use --robustHesse=1
If nothing works, it could be a good idea to try to inspect the problematic uncertainties. I don’t remember if you need to increase the verbosity for this, but In the output of the combine command it should print the final value of the parameters. If any of them looks odd or on the boundary you can try freezing them and see if that was the problem, and afterwards we can think about how to put them back in the fit.

Let me know how these tests go
Best,
Giacomo
