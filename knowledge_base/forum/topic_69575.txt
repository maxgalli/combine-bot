REPLY 2 (by nckw @ 2024-11-27T19:12:40.000Z):
Hi Vinay,
Just to be clear, you are trying to calculate the covariance between bins in your distribution right, as opposed to between the parameters of the model?
If the former, FitDiagnostics will sample from the covariance for the parameters to calculate the covariance of the bins, I suspect it is simply the process of looping over all the bins (squared since it’s the covariance) across all channels that is taking so long per toy (this part of the code is really not very smart in all honesty) - you could try reducing the number of samples (see the docs) just to check if that makes things faster.
Another approach is to run the toys yourself and keep track of the bin contents in each fit (you’ll find options for that too) this has the bonus that you can then parallelise the jobs to do so but the downside is that you’ll probably have to write a bit of code yourself to calculate the covariance from the toys after. If you happen to write something generic, then it would be great to add to the set of tools of course
Best
Nick

REPLY 4 (by nckw @ 2024-11-28T19:03:31.000Z):
Hi Vinay,
To be clear, when I say reduce the number of samples I mean to reduce the number of toys combine internally throws to calculate the covariance using the option --numToysForShape
For the toy generation yourself, your command is ok but you need to include the option --savePredictionsPerToy as mentioned here https://cms-analysis.github.io/HiggsAnalysis-CombinedLimit/latest/part3/simplifiedlikelihood/?h=moment#requirements
Once you do, you will find branches inside the trees “n_exp_*” which gives the number of events in each toy per process, per bin (for details on the naming, see here : https://cms-analysis.github.io/HiggsAnalysis-CombinedLimit/latest/part3/nonstandard/#per-bin-norms-for-shape-analyses
What you need to do is sum over the background processes in each toy to get the total background in each bin for each toy. From there you can calculate the covariance between bins.
The tree you want depends on the type of analysis. If you have no control regions, you can simply use the prefit tree, if not then you would need to mask the signal region as usual and use the fit_b tree to get what you need.
Hope that makes sense,
Nick

REPLY 7 (by nckw @ 2024-12-10T15:49:43.000Z):
Hi Vinay,
These two are not expected to be the same. The reason is that in [1], you are directly re-fitting each toy to find the yield (it really is the estimator in each toy), and this is actually more correct. In [2], we use a short cut by sampling from the post-fit covariance matrix which makes some assumption about the Gaussianity of the post-fit nuisance uncertainties & correlations – its good that they are similar, but it doesn’t have to be the case.
Then when calculating the moments, in [2], they are calculated wrt the data fit as the “expected value”, while I suspect in [2], you again calculate the moments properly.
To get the total background, you should just sum up the different processes (you could either use the average of the toys or the one directly from the fit, though they can be different, and you may find that with the latter one, your “post-fit” results using the simplified likelihood looks more like the full combine ones).
In the end, You can check that you get a similar result (say when fitting the signal strength) using the simplified likelihood compared to combine – there is an example at the end of the instructions to do that.
Remember, all of this is an approximation and even what we mean by “uncertainty on the background templates” is an approximation so its not surprising that different assumptions on what these are can lead to different results (slightly).
Best,
Nick

REPLY 9 (by nckw @ 2025-01-21T20:29:34.706Z):
Hi Vinay,
Thanks for sending these studies. It looks very convincing to me so I would say in this case the toys-based approach is certainly making sense. It would be great to include instructions (and even the scripts) in the combine repo to allow for this option to be used instead of the default one.
For the case where there are no control regions in the datacard, essentially what combine does is throw nuisance values from the nuisance pdf and re-evaluate the yields (since there’s no data to account fot). To do this with the same approach as you used for your current study, you need to make sure the toy generation is done in the same way.
It should be enough to use -t N and check the yields from the pre-fit branch in the fitDiagnostics.root file. You should not use the --toysFreqentist opeion as this will instead sample the observables, which means each toy will have an identical PDF (and hence yield in each bin). So your command is fine, just remove the --toysFrequentist option.
Hope it makes sense,
Best,
Nick
