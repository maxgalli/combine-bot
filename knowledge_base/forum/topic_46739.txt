Dear Experts,
I have a question about the " number of iterations" parameter in MCMC method. I am estimating the Bayesian limits with MCMC method. The physics model I used is attached below [1].  I have a pure signal term, an interference term and DY term in the signals as I am calculating limits for EFT based interference signal scenario. A few more details are listed below:
Number of channels: 8 for Full Run 2 per b jet multiplicity, and I have 0b, 1b and >1b categories.
Sample datacard is attached below [2].
combine command:
combine -M MarkovChainMC dataCards_mumu_2LB_negLL_obs_ZeroB/mumu_2LB_combined_6_negLL.root -n mumu_2LB_negLL_obs_ZeroB -m 6 -i 50000000 --tries 6 --prior flat --rMax 10
As you can see I have increased “number of iterations” to 50M, but I am not getting any output or limit value in the end.

Random number generator seed is 123456
Method used is MarkovChainMC
Done in 1078.32 min (cpu), 1081.91 min (real)
Finished – ls dir:

I tried running the same with 2 channels only, I got the output with “number of iterations” = 5M. The output looked reasonable too.
For other signal scenario, I have observed that yes, one has to increase the "“number of iterations” value to get stable limits but for other signal scenario, I reached plateau at 3M.
My question is: Is this >50M “number of iterations” common in practice and is safe or we have to fix some other parameter to get the results. I tried changing the number of tries from 6 to 100, but no luck.
Please let me know if there is any solution to this problem.
Thanks and regards,

REPLY 2 (by kcormier @ 2024-08-27T10:45:06.504Z):
Dear Amandeep,
I think you forgot to provide the root input files that are inputs to the datacard, so I can’t check in your particular case. But here are a few general points for you to check and consider in the meantime.
I think if no limit is being provided, it probably means that the number of points accepted in your chains is less than the number of burn in points (that should be thrown away) for all of the tries. We should probably provide more clear terminal output of this. You should be able to check this by running with -v 1, and you may see messages like:
[#0] ERROR:InputArguments -- MCMCInterval::CreateVector: creation of vector failed: Number of burn-in steps (num steps to ignore) >= number of steps in Markov chain.

You should also see the number of accepted points in each try and the acceptance rate. I suspect that what is happening is that you have a very very small acceptance rate (default number of burn in steps is 200, so if you are using 50M iterations, something significantly less than 1/25k ).
If this is the case, increasing the number of iterations is probably a very bad strategy for getting it to converge – it suggests that there is a problem with the proposal, i.e. your proposal algorithm is proposing almost exclusively points which have almost no probability of being accepted.
In combine, you have a few handles on proposal (see the docs [1]). But the default proposal algorithm (ortho) which varies one parameter at a time is typically higher acceptance than the gaussian or uniform ones. So it seems likely that the step size is too large, by default this is set to 1/5 the parameter range, so if your likelihood function varies quite rapidly in the parameters with respect to 20% of their range, you can end up in such a situation. You can adjust the value of the range/step size ratio using --propHelperWidthRangeDivisor <ratio>.
I’m not really an expert on this, but I suspect that perhaps the issue in your case is related to your POI specifically, since by default the proposal function will always step the POI. If the range of the POI is much larger than the width of likelihood that could explain the issue. You could test this very easily by running with the option --alwaysStepPOI 0 and checking the behaviour of the acceptance. This will generally have worse convergence, but if that helps, you could then pick a more plausible range for your POI (by checking with e.g. MultiDimFit).
Best,
Kyle

REPLY 4 (by kcormier @ 2024-08-27T15:18:47.473Z):
Hi Amandeep,
The number of steps in the chain is just the number of accepted proposals. The number of iterations is the total number of proposals. i.e. your markov chain made 100 proposals, of which 1 was accepted.  To increase the number of accepted points, you can just increased the number of iterations, but to change the acceptance rate you need to change something about the proposal mechanism.
Best,
Kyle

REPLY 7 (by gortona @ 2024-09-24T08:35:07.000Z):
Dear Amandeep,
In principle, a negative signal PDF is not a problem as long as the sum of signal + background is positive in all bins.
There is no built in strategy in Combine to enforce this requirement. It is up to the users to ensure that their model is well-behaved as there are simply too many cases to take them all.
The first things to check are usually to restrict the ranges of the POIs and of the nuisance parameters in order to force the fit to operate in well-behaved regions.
Although the distribution you sent is a bit puzzling: it seems to me all the bins except the negative one are set at exactly 0. This configuration seems a bit unrealistic. I’m not sure what is going on, but I think it would be better to have a careful look at the inputs to the workspace.
I hope it helps,
Best regards,
Giacomo
