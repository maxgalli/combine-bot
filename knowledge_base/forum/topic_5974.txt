QUESTION (by svanputt @ 2022-02-08T11:59:42.174Z):
Dear combine experts,
I am trying to combine two different run 2 channels, one semi-leptonic decay and one full-leptonic decay.
In order to decouple the trigger nuisance, I was trying to rename it using the
nuisance edit rename process channel oldname newname [options]

strategy, when I came across some unexpected behaviour.
I combine the 2016 samples for the full-leptonic case in datacard_2016_full_lep_1.txt with combineCards.py and append the line
nuisance edit rename * * CMS_eff_hwwtrigger_2016 CMS_eff_2l2nu_trigger_2016 ifexists

Next I recombine it to a new card
combineCards.py datacard_2016_full_lep_1.txt > datacard_2016_full_lep_2.txt

The combine code altered the rename command with the explicit coding
nuisance edit  rename (DH_mhs_160_mx_100_mZp_500|ggWW|WWewk|Higgs|VVV|top|DY|WW|Vg|Fake|VZ|VgS) ch1_(Year2016_TCR|Year2016_DCR|Year2016_WCR|Year2016_SR1|Year2016_SR2) CMS_eff_hwwtrigger_2016 CMS_eff_2l2nu_trigger_2016 ifexists

Which looks fine to me. Except when I now fit the model with FitDiagnostics I receive fit values for both CMS_eff_hwwtrigger_2016 and CMS_eff_2l2nu_trigger_2016.
I tried to do a similar thing with the semi-leptonic datacards, renaming CMS_eff_hwwtrigger_2016 to CMS_eff_lnujj_trigger_2016.
But there the behaviour is as expected and I only get fit values for CMS_eff_lnujj_trigger_2016.
So it sounds like there would be something misaligned in the full-leptonic datacards, except everything looks normal and the full-leptonic and semi-leptonic datacards have been made with the same framework.
Can you help me locate the issue?
I put the examples in /afs/cern.ch/user/s/svanputt/public/for_CMStalk/combine/ to have a better overview of the exact commands that have been used.
Cheers,
Senne

REPLY 2 (by adewit @ 2022-02-11T13:10:44.880Z):
Hi Senne,
It looks like the rename does not happen for the VgS process because the regex match for
nuisance edit  rename (DH_mhs_160_mx_100_mZp_500|ggWW|WWewk|Higgs|VVV|top|DY|WW|Vg|Fake|VZ|VgS) ch1_(Year2016_TCR|Year2016_DCR|Year2016_WCR|Year2016_SR1|Year2016_SR2) CMS_eff_hwwtrigger_2016 CMS_eff_2l2nu_trigger_2016 ifexists
Finds a match with Vg first -  and does then not check for a match with VgS anymore. Since we only accept full matches the regexp match with Vg is not of the correct length, and so the command does nothing for VgS. This is obviously not intentional! In any case, the simplest thing you can do is manually re-order the command so that VgS comes before Vg. Or you can edit it to use the asterisks like in your original card. Both should have the same effect.
Best,
Adinda
