REPLY 2 (by adewit @ 2023-01-12T14:01:48.995Z):
Hi Sasha,
It’s possible to run fits like this without a signal process by passing --X-allow-no-signal. It’s probably easiest to run text2workspace manually, rather than letting combine call it under the hood. So for example:
text2workspace.py 2017ul_dimuon_SR1_FTEST.txt --X-allow-no-signal -o ws_2017_sr1.root
Then you can run combine -M GoodnessOfFit --algorithm=saturated -d ws_2017_sr1.root
Before you do this however, there are a few of problems with the datacard that you need to fix, otherwise the fit will not run for other reasons:

The shapes lines at the top are missing an entry for the bin label, it should say

shapes data_obs Firstbump_mumu  2017ul_RooFitWorkspaceSR1_FTEST.root w:data
shapes bkg      Firstbump_mumu  2017ul_RooFitWorkspaceSR1_FTEST.root w:bkg

(or you can use * instead of Firstbump_mumu)

The background rate cannot be set to -1, it’s not possible for combine to automatically determine the number of background events since bkg is just a pdf. I would set it to 1, then you can introduce a rateParam for the background normalization, by adding at the end of the datacard:

bkgnorm rateParam Firstbump_mumu bkg 300 [0,3000]

(or replace the nominal value of 300, and/or the fit range given in brackets, by something else. I took these numbers from the ‘norm’ parameter that is in the input workspace.
Hope this helps,
Adinda

REPLY 8 (by adewit @ 2023-01-18T08:43:10.146Z):
Hi Sasha,
Apologies for the slow reply. It would be best if you can make a histogram of the toy distribution, so you can see if it is healthy (then you can also calculate the p-value as discussed here: https://cms-analysis.github.io/HiggsAnalysis-CombinedLimit/part3/commonstatsmethods/#goodness-of-fit-tests). One thing you may also want to check is whether the initial values of the parameters of your function are the best fit values that you’d expect, or not. If not, this could lead to some discrepancies.
Best,
Adinda
