Hi Chris,
You’re right that the uncertainty in the ttbar rate param should be bounded by the luminosity uncertainty. However, the luminosity parameter and the ttbar rate param will be anti-correlated postfit, such that the uncertainty in the total prediction usually corresponds to the data uncertainty in similar cases. In other words, that the total uncertainty can be much smaller than, say, 1%, doesn’t mean that we know the rate param (or the cross section) at the 1% level.
With that in mind, I think the question makes perfect sense. One thing I’m wondering is if looking at ttbar is good enough, since what I write above is in general only true for the sum of backgrounds. So maybe there’s some anti-correlation with the W+jets background and you need to look at all backgrounds combined?

Aliya’s observation tallies with what I saw yesterday. Just to note that the uncertainties for the individual process contributions can be larger than the total uncertainty as a result of anticorrelations between nuisance parameters. The question of exactly what’s plotted is therefore very important, and in particular how the uncertainty band is constructed. If you are adding up any of the uncertainty bands from plots in the fitdiagnostics.root file the band will be too large, as you end up missing correlations between the parameters.

REPLY 9 (by adewit @ 2023-03-03T13:55:45.911Z):
Hi Dennis,
For your first question, I don’t know if anyone else has a more quantitative answer, but due to the presence of shape-altering uncertainties I don’t think it’s entirely unexpected that the total uncertainty on the prediction doesn’t exactly equal the data uncertainty in each bin.
For the second question: yes, it can be taken into account. At the moment the easiest way is probably to use PostFitShapesFromWorkspace, a tool in CombineHarvester: http://cms-analysis.github.io/CombineHarvester/post-fit-shapes-ws.html
In addition to the options shown in the example usage on that page I would use --skip-prefit  --skip-proc-errs  --total-shapes  to speed up the the process (the two skip options) and to get shapes for categories summed together (total-shapes).
In your case you don’t want to add all the categories together, so you should extract from the datacard only the categories that you want to combine into a single plot (e.g. the bbww_sl__201*_all_incl_sr_prompt_wjets_other for each of the years). The datacard that you pass to PostFitShapesFromWorkspace should only contain the categories that you want to combine. The workspace that you pass should also correspond to this datacard. However, the fit result you pass with -f will be the fit result from the combined fit (ie from the fitdiagnostics file that you shared earlier).
One thing to keep in mind: make sure that the category names for the categories included in the “reduced” datacard are exactly the same as in your fully combined datacard, otherwise uncertainties from autoMCStats will not be propagated correctly.
This process should then be repeated for every set of categories you want to combine into a plot. I agree this is a little cumbersome - I believe there is a plan to develop PostFitShapesFromWorkspace to be able to do this kind of category combination a little more easily, though I’m not sure anyone has started working on that yet.
Hope this helps,
Adinda

REPLY 15 (by kcormier @ 2023-03-08T20:39:26.863Z):
Hi Christian,
The sampling is done by the RooFitResult, using the randomizePars method [1]. This takes the full covariance matrix into account. Although it doesn’t do any eigendecomposition, it is explicitly building the correlated random normal vector by starting with a standard random normal vector, and then multiplying it by the matrix A, where AA^T is the covariance matrix, and then adding any necessary offset. It constructs A using the “square root method” from the covariance matrix.
[1] https://root.cern.ch/doc/master/classRooFitResult.html#a6b9356808af06ab6c91bc8b07918f2f6

REPLY 17 (by kcormier @ 2023-03-09T12:21:37.454Z):
Hi Christian,
No problem. Understandably these things are not all obvious. Here is a short (but incomplete  and unrigorous) argument that its doing exactly what the covariance matrix should. Maybe it at least gives a little intuition.
You want a distribution of random vectors, v,  sampled such that  v^T Cov^{-1} v follows an n-dimensional chi-squared distribution. You construct the vectors by doing v = An  where n are vectors sampled from a standard n-dimensional normal distribution. so:
v^T Cov^{-1} v = n^T A^T Cov^{-1} A n =  n^T A^T (AA^T)^{-1} A n = n^T n
which is sampled from an n-dimensional chi-squared distribution, as intended.
There are probably better ones, but thats the first argument that came to mind. You could probably play around with the math for a little bit to get a fuller explanation; and  I’m sure there are more academic resources, but this is also how wikipedia defines their random normal vector [1]. If you want more resources about the “square root method” to build the triangular matrix, it is explained on wolfram [2], but this is just about the equation to derive A from the covariance.
As for which correlations are taken into account, there are of course limitations of the covariance matrix itself, but these will be true no matter what you do with the covariance matrix, as it simply does not contain all information. It is giving you information about linear correlations around a given point, which is fine assuming that the likelihood shape is approximated by a multivariate normal. This should be approximately true at an optimimum, since it would be the leading term in the taylor series expansion, hence the connection between the coaviarance matrix and the hessian.
So it will not take into account other forms of correlation that can come up due to more complex shapes of the likelihood, particularly non-linear correlations between parameters. But this is true even with two parameters (consider e.g. a likelihood shaped like the higgs potential in 2D).
So there are indeed things that this will miss, but I would be surprised if this was the source of your particular issue. In particular, correlation matrices (such as those on s.8 ) also show the same linear correlations. I assume that this particular correlation matrix was in fact constructed in the same way as the covariance matrix. So it would be surprising to me if it contained information showing that the covariance matrix is insufficient, and the full covariance matrix should be taken into account in the sampling.
But I’ve been surprised before. That’s just my two-cents without having looked in detail at your fit setup.
I hope its helpful,
Kyle
[1] https://en.wikipedia.org/wiki/Multivariate_normal_distribution#Normal_random_vector
[2] https://mathworld.wolfram.com/SquareRootMethod.html

Note that the 12450 value here is a fair bit lower than the number 14068.3 from the README. This is because the uncertainty estimate used in CombineHarvester’s PostFitShapesFromWorkspace tool is using a slightly different calculation than in FitDiagnostics (see the code here, basically instead of subtracting the sample mean it subtracts the best-fit values.)
The reason why these toys are not accurately reflecting the data constraint is because the multivariate gaussian approximation is rather poor for this model. If we scan the NLL of the true model and compare with the fit result’s approximation it’s rather obvious:

(the contours are too close together to see easily)
One thing we could do to correct this in principle would be to reweight each toy by the ratio \mathcal{L}_{full}(\theta) / \mathcal{L}_{gaus}(\theta) where the latter is just \exp(-\theta^\top C^{-1} \theta/2)
Doing this, we mostly recover the distribution of yields we would expect, and what we see when using the gaussian uncertainty model (since there the correlation between the nuisance parameters is more linear)

But we’re playing fast and loose here: the likelihood is not a probability distribution, and neither is its ratio. The construction above is mathematically equivalent to sampling a Bayesian posterior for flat prior in this situation, interpreting s1 and s2 as parameters of log-normal distributions with observations fixed to ln(1)=0, which I would hypothesize is why it still works well.
Another way to correct this is to reparameterize the likelihood with the substutution t_i = 1.4^{s_i} which changes \mathcal{L} = \text{Poisson}(n_{obs}|n_1 1.4^{s_1} + n_2 1.4^{s_2}) \text{Normal}(0|\mu=s_1, \sigma=1)  \text{Normal}(0|\mu=s_2, \sigma=1) to \mathcal{L} = \text{Poisson}(n_{obs}|n_1 t_1 + n_2 t_2) \text{Normal}(0|\mu=\ln(t_1), \sigma=\ln(1.4))  \text{Normal}(0|\mu=\ln(t_2), \sigma=\ln(1.4)) which does not change the model since we are free to reparameterize the likelihood. In the case where the observed count is very high (here 1e6), the Poisson term dominates the NLL, and hence in the t-parameterization the NLL is better approximated since the anti-correlation in t is more linear:

The reparameterization would help in this case, but on the other hand it would hurt in the more typical case where the data does not constrain strongly the nuisance parameters. In that case, the shape of the NLL (with respect to the nuisances) is more driven by the constraint term, so the s-parameterization is better approximated by a second-order expansion.
I think this highlights the fact that, although we are free to re-parameterize the nuisance effects however we like without changing the maximum likelihood estimate or the profile likelihood ratio used to define confidence intervals on POIs, these parameterizations can affect the validity of the approximation of the full likelihood near the minimum. At the end of the day, the only truly robust frequentist intervals would be made by throwing toys in the observables with the parameters fixed to their post-fit values (-t N --toysFrequentist) and then fitting each and building the distribution of expected values for each bin. That’s a lot of computing just for error bars on a plot. (anyway, the bar is low  for quality error bars)
All of the plots can be made from this notebook, to be run in the postfit-band-test area.

REPLY 28 (by ncsmith @ 2023-04-10T15:22:59.414Z):
I think the most straightforward way to generate and re-fit toys is to do something like the following:
for i in {1000..1199}; do
combine -M FitDiagnostics lnN.root --skipSBFit -n .lnNtoy.seed${i} \
--saveShapes --saveOverallShapes --expectSignal 0 --toysFrequentist -t 1 -s ${i}
done

and then load each shape or norm and compute error bars from the distribution of those best-fit yields per toy, e.g. with
def getyield(toy):
fit = ROOT.TFile.Open(f"fitDiagnostics.lnNtoy.seed{toy}.root")
return fit.Get("shapes_fit_b/total_background").GetBinContent(1)

lnNfromToys = np.array([getyield(i) for i in range(1000, 1200)])
print(f"lnN from toys total yield: {lnN_yield[0]:.3f} ± {np.std(lnNfromToys):.3f}")
