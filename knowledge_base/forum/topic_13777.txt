QUESTION (by lguzzi @ 2022-08-09T13:44:59.043Z):
Dear combine and stat experts,
I am working on the \tau\to3\mu analysis, which looks for the BS \tau\to3\mu process as an accumulation of events in the 3-muon invariant mass distribution. The result of the analysis will be an upper limit on the \tau\to3\mu branching fraction.
At the moment, I want to compute expected limits based on the expected number of background and signal events.
The number of expected signal events is computed on a MC signal sample, the number of expected background events is computed fitting the 3-muon mass sidebands and taking the integral of the fitted pdf (an exponential) in the full range, as shown in [1].
The datacard is then built as shown in [2]. Different contribution to the signal normalization and shape are treated as lnN-constrained nuisances. A free parameter regulates the background shape (a0_C18 in the datacard) and another regulates the background normalization (bkgNorm_C18 in the datacard). The observed number of events is set to -1.
Signal normalization nuisances are 0 by construction (that is, the signal normalization is 1 -  in facts, the MC is normalized with external weights), the signal shape is initialized by the MC signal sample, and the background shape is initialized by a fit to the signal sidebands, as already shown in [1].
Now, I want to compute the expected upper limit using the LHC test statistics w/ toys (CLs). To my understanding, expected means that the nuisance parameters are not re-fit on the observed data before generating the toys (that is, the background normalization is 1, so that each toy behaves as an un-blinded dataset where the observed number of background events is equal to the expected one modulo fluctuations). Is this correct?
The workspace connected to the datacard contains the “data_obs” RooDataSet, which is the set of data events passing the analysis selection, blinded in the signal region (meaning no events from the signal region are put in the dataset). If my understanding of exp. UL above is correct, this should not matter in the expected limit computation as this sample is not used.
I compute the UL with the following command
text2workspace.py datacard.txt -m 1.777 -o workspace.root -D data_obs
combine -M HybridNew --testStat=LHC --fitNuisances=0 --frequentist workspace.root -T 1000 --expectedFromGrid 0.5 -C 0.9 --rMin 0 --rMax 10 --setParameterRanges $PAR

where $PAR just limits the signal width between 5 MeV and 100 MeV, the background slope between -100 and 100 and the background normalization between 0 and 10^5, only to help the fit convergence. In this command, as far as I understand, --fitNuisances=0 would be what makes the limit “expected”.
However, from the combine log (using the -v2 verbose argument) I see that for each scanned value of the signal strength the dataset “data_obs” is re-fit. See for example [3]  at line 269 (ctrl+F  0x7cc7610 RooRealVar::            r = 3.68898).
Indeed, if I save in the datacard workspace the full set of observed events (i.e. data_obs is not blinded), the limit changes. This suggests that something is not correct in my strategy.
Could you please help me understand where the mistake is?
thank you,
best,
Luca

REPLY 2 (by adewit @ 2022-08-09T14:02:43.307Z):
Hi Luca,
I think the option --frequentist sets --fitNuisances=1, and it looks like this option gets resolved after the command line --fitNuisances=0.
In any case, I would recommend to follow the suggestion discussed here [*] under “Warning” - that is, to generate a pre-fit asimov dataset and use it in place of the data, then run HybridNew as recommended (with --LHCmode LHC-limits). This seems to be what you would like to do, and avoids any unexpected interplay between the different options.
Best,
Adinda
[*] https://cms-analysis.github.io/HiggsAnalysis-CombinedLimit/part3/commonstatsmethods/#computing-limits-with-toys

REPLY 7 (by amarini @ 2022-08-10T09:15:07.000Z):
Hi,
I think it is always a good idea to proceed as follow:

Determine that the fit is unbiased in s=0 and s<>0
Use the asymptotic formulae to understand the limit feature
eventually use toys to understand particular regions or validity of the asymptotic formulae.

As long as you are blind the asymptotic limits are good enough for your scope. You can test it on particular datasets, where in the blind region you have no events (for instance).
Cheers,
-Andrea

REPLY 8 (by adewit @ 2022-08-10T09:45:29.841Z):
Hi Luca,
To answer your original question: there is always only one “data_obs”, whether it’s real data or a toy dataset. When generating post-fit expected limits with the real data we do not change the data set when we scan through different signal hypothesis points (internally in the toy generation that HybridNew is doing there are of course toys for different values of r, but this is the case regardless of whether you have used the real data as data_obs or an asimov dataset). The way to generate blind expected limits with HybridNew is as I suggested, replacing data_obs with a backround-only asimov dataset simulates the case where the data follow the background model exactly. This is the same assumption as when running blind asymptotic limits.
As Andrea said, you can of course use AsymptoticLimits for the moment (while the analysis is blinded), you may find this procedure simpler (not having to generate a dataset to replace the data) and faster.
Best,
Adinda
