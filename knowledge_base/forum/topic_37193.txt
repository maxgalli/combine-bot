REPLY 2 (by kcormier @ 2024-03-25T15:31:45.591Z):
Hi Gianfranco,
There are a couple of not-so-obvious things going on here.
When the significance is calculated by HybridNew, the pvalue is calculated based on the observed test statistic and the distribution from toys. The p-value is calculated using the integral of the distribution, with the observed test statistic value as one of the limits of integration, and it is calculated using this as a closed interval. I.e. the observed test stat statitic value itself is included in the integration.  So if the observed value of the test statistic is = 0, you’re going to get a p-value of about 1/Ntoys, and a corresponding significance.
This is a little different from whats done in the asymptotic case, where we are implicitly using an open integral for the p-value, such that an observed value of 0 gives a significance of 0, rather than -infinity.
So I don’t think the results you are getting are inconsistent, though I understand that it’s rather confusing. You could run the asymptotics with --uncapped 1 to allow negative significances, but to do the equivalent thing with toys you’d also need to change the test statistic to remove the condition for \mu < 0.
I also noticed quite a large number of failing fits when running the toys, which you might want to follow up on.
Concerning the autoMCStats:
The tracked parameter you point to is actually a gaussian constrained parameter for the entire bin, not a parameter for only the signal process. You can tell because the name contains identifying information about the bin only and nothing about the process.
Because you’ve set the threshold value in your datacard to 0, the autoMCstats method wants to treat every bin with more than 0 effect events using a single gaussian constrained parameter which will multiply the yields of all processes [1]. So, it’s possible for this parameter to get pulled regardless of whether any signal is present or not. In order to have a single parameter defined per process you need to set the threshold high enough so that each bin is below that number of effective events. Then processes with no uncertainty are ignored, all other process will have individual nuisance parameters created for them (which may be gaussian or poisson).
If I edit your data card accordingly, then  checking a few parameters they aren’t pulled in the r=0 case. Note, however that my out-of-the-box test of this didn’t return sensible fit results (for example, the best fit value of r hit the boundary at 20).  Also, in the datacard as you have it set up, there seem to be some pathologies. You can check how the constraints are being said when you run text2workspace.py on the datacard, and you’ll see even with your threshold at 0, some of the bins are getting poisson constraints, apparently because the effective number of events appears negative, but I haven’t followed up on exactly where this is coming from.
I hope that helps,
Kyle
[1] https://cms-analysis.github.io/HiggsAnalysis-CombinedLimit/part2/bin-wise-stats/

REPLY 4 (by kcormier @ 2024-03-25T21:45:00.269Z):
Hi Gianfranco,
Indeed, for your case using GammaN is probably just as easy, or easier than using autoMCstats in this somewhat unusual way. It might be worth following up on the negative events in any case, I worry that there was something strange going on in the model, that was just showing up in that spot and was not that the autoMCstats was the problem – but I didn’t follow up to find the cause, so I could be mistaken.
For the Hybrid New:
It doesn’t necessarily mean that the toys have positive best-fit signal strength.  If all you want to do is remove the condition on setting the test statistic to 0 if the best fit is negative, you can just use the profile likelihood test statistic directly, rather than the modified one which is normally used with --testStat Profile.
If you want to check the distribution of the test statistic values (to see e.g. if many of them are 0) you can plot the distribution with the script test/plotTestStatCLs.py. e.g.
python test/plotTestStatCLs.py --input higgsCombineTest.HybridNew.mH120.-568846852.root --poi r --val all --mass 120 --signif

on the output file from HybridNew, that will create a root file with a plot of the test statistic distribution, which you can check.
But I don’t think that’s your issue. Consider the case of performing the p-value calculation using the closed interval  [ q_{obs}, \infty].  If q_{obs} is equal to the lower limit of the domain of your test statistic, this will be 1, and your significance will be given by the corresponding inverse quantile function of a normal distribution, so -\infty.
In the asymptotic case, for the LHC-significance test statistic the expected distribution is a mixture of \delta(0) and the positive side of a gaussian. if q_{obs} is 0 and you do the closed interval integral, the significance would be -\infty. But, when you run the -M Significance method, it implicitly uses an open interval and so the p-value is 0.5 (because \delta(0) is excluded) rather than 1 and the significance is 0. You can also see this if you run with --pvalue when running your asymptotic signfiicance command.
In other words, this is a bit of a funny case only because it depends on where you include the value of q_{obs} in your integral or not, which happens because there is a finite probability density in an infinitesimal volume for these expected distributions. And HybridNew and the asymptotic case use different conventions for this.
But you can run --uncapped for the asymptotic calculation and --testStat Profile for the HybridNew calculation to check the p-value under the normal Profile Likelihood test statistic, where this kind of subtelty shouldn’t occur.
Best,
Kyle
