For adding nuisance parameters to a parametric model, you can have any parameters constrained to a particular value + uncertainty (with the param keyword). You can see the description here.

Why are the errors on the fit values so large?
my first guess is that for the reason above you may have instabilities. The error on the fit is the band which is obtained from resampling toys. This depends on the parameters you have and they interplay.

When I change the fit range for the RooRealVars it also changes these errors bars and the visualization band in the plot.

RooRealVars freely floating are constrained by the observation and allowed to vary only in their range. If the value of the fit would try to probe that variable outside the given range it stops.  This is why the errors bars changes. It may not be correct to force the fit in a particular region.

Another iteration I try with a RooPolynominal without squaring the coefficients. There are plenty of warnings showing that the normalization goes negative during the fit.

The fit is not bounded to be in the positive range, unless there are events in the observation that protects this to happen. Unfortunately, probability density functions are define only on the positive domain.  I saw in the past squaring the coefficients of the bernstein representation of the polynomial, but not of the polynomial themselves.

Hi,
All the generalizations that I specify below, may be really situational to the details of the analysis.
For implementing a polynomial (or other roofit functions), background (pyroot):
# load combine library, if you used pdfs defined in combine, eg RooBernsteinFast
ROOT.gSystem.Load("libHiggsAnalysisCombinedLimit.so")
# declare a workspace
w = ROOT.RooWorkspace("w","w")
#make a variable and get it
w.factory("xobs[130,1000]"); # RooRealVar
xobs=w.var("xobs")
## or create a variable and import it in the ws
## xobs=ROOT.RooRealVar("xobs","xobs",130,1000)
## getattr(w,'import')(xobs,ROOT.RooCmdArg()) ## import is a python keyword.
#define a set of coefficients and the pdf for each category in your fit. Each with a different name.
coeff = ROOT.RooArgList("bern_coeff_catX")
for i in range(0,order+1):
b = ROOT.RooRealVar("bern_catX_%d"%(i),"bern_coeff",0.1*(i+1),-10.,10.)
## values and range are completely arbitrary here. They depends on your shape, order, normalization and sensitivity.
coeff.add( b ) # if you want them squared, make a RooFormulaVar that squares them here, and add them instead
g.extend([b]) # make sure it is not deleted by pyroot
# this is templated on order
bern = ROOT.RooBernsteinFast(order)("pdf_bern_catX","bernstein",xobs, coeff)
#create a normalization for the pdf
bern_norm = ROOT.RooRealVar(bern.GetName()+"_norm","",data[cat].sumEntries(),data[cat].sumEntries()*.5,data[cat].sumEntries()*2.0) # put a reasonable range, x2,/2 of the data entries is mostly fine as long as you start to have data (N>3) in the category.
#if you want reasonable starting value to have the prefit Asimov somehow meaningful, you can start to fit to each category to get a starting point. This will also help the fit to converge. for this purpose a model.fitTo(data) with some option should be enough here.
#import in the ws
getattr(w,'import')(bern,ROOT.RooCmdArg())
getattr(w,'import')(bern_norm,ROOT.RooCmdArg())
## you may want also to create and import the RooDataHist on the same observable, if you have a TH1 the following code should work.
arglist_obs = ROOT.RooArgList(xobs)
roo_data= ROOT.RooDataHist("data_obs_catX","xobs",arglist_obs,h_data)
getattr(w,'import')(roo_data,ROOT.RooCmdArg()) ##
## RooDataHist -> corresponds to binned likelihood.
## RooDataSet (from a tree) -> unbinned (you will need to be careful in the definition of the asimov). My advice is to start to set up the binned fit, and later if needed try to switch to the unbinned one.
## notice that the signal as well will need to be on the same obs.
## for the signal (and the bkg) remember that all the things that you make float in the workspace, they'll float in the fit, and all the things that you declare constant will be constant. In general, (but this may be situational), all or most of the background parameters should be floating, and all the signal related ones should be constant or constraint to some value (except r, that is added by combine).
# save the workspace
w.writeToFile("workspace.root")
Your datacard will contain the sig, bkg, and data specifications, and combine will construct the likelihood for you of the form:
L = L ( r*S+B|data)
On the one side impacts, double check the model first. If you expect that a variation on a parameter changes the yields in the same direction, than you may have the one sided impact. (Eg, for squared coefficients if they fit around 0). All other cases, may be suspicious and should be investigated/understood.
Let me also add that we have contacts for each PAG that should also help you constructing the datacard. https://twiki.cern.ch/twiki/bin/viewauth/CMS/HiggsWG/HiggsCombination
