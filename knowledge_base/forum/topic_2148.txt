Dear combine experts,
For the computation of unblinded pulls and impacts for the HH->bbtautau
analysis (HIG-20-010), we would like to reduce the runtime and increase
the stability of nuisance parameter fits. In particular, we aim to find
a suitable set of minimizer options (mostly tolerance and fallbacks) for
the initial fit, based on the smallest NLL value, and then use the saved
workspace as a snapshot for the nuisance impacts.
So far, we achieved that by using commands [1] and [2], which result in
both pulls and impacts looking reasonably well. Now, we were wondering
if we also have to use the "--skipInitialFit" option for [2] to avoid
rerunning the global fit based on an already optimized set of workspace
parameters, and to improve the overall performance. However, when using
that option, the impacts are actually never calculated since "res" in
[3] will always be a nullptr in this case.
Therefore, we would like to ask if it is considered safe to load a
snapshot but not use the "--skipInitialFit" option (stability and
runtime still improve since the fit starts from good initial
parameters), or if it is not recommended to use a snapshot for pulls and
impacts in the first place.
We are using combine v8.2.0 inside a CMSSW_10_2_13 environment.
Thanks a lot in advance!
Best regards,
Marcel for the HH->bbtautau analysis team

Hi Konstantin,
You don't say how you're extracting the NLL values from the workspaces,
so I can't say much about those, but in general if you want to track the
absolute NLL values it's best to use --saveNLL when you run combine,
together with --X-rtd REMOVE_CONSTANT_ZERO_POINT=1 so that you get the
"true" absolute NLL values (otherwise constants are always subtracted so
that you can't easily compare these results when you're fitting
different workspaces like this)
The fact that the deltaNLL in the second step is just below 0 indeed
means that the second fit finds a slightly better minimum. I think it's
expected that this can happen, the minimizer tolerance is finite after
all. You can check how different the fitted parameters are and see if
there are any very significant shifts in any of the parameters that you
wouldn't expect.
Best,
Adinda

Hi Konstantin,
Yes, you can compare nll+nll0 between the two runs. Looking at your
commands again I see that you are using --setParameters r=1, this is the
value used in the step 1 "nll0" evaluation, rather than the best-fit r
from step 0 (which is presumably not 1).
To me all of this looks consistent; you are finding a minimum at the
same nll+nll0 value in both cases. Of course, my previous comment about
the second fit finding a better minimum should be disregarded (the nll
difference from nll0 seems to just be due to the initial value of the
POI, as evidenced by the consistency of the absolute NLL values you
stored in the combine run).
