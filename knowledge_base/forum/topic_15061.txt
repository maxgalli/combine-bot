The plot of VH in full is here, using the default Minuit2 options (and MinimiserStrategy 1):
original_scan_default_minuit2.pdf (17.3 KB)

fix_scan_GSLMultiMin_vh.pdf (17.3 KB)
The best fit value in both cases is mu = 0.22, with the unc changing from +0.26 -0.22 to +0.23 -0.30.
We intend to update our PAS and paper with this new plot (short thread on this here: https://cms-pub-talk.web.cern.ch/t/lh-scan-in-vh-categories-fix-with-gslmultimin/7174), but wanted confirmation from the HComb committee that this procedure is indeed appropriate for a final result? REPLY 2 (by agilbert @ 2022-09-16T09:18:37.206Z):
Hi Rob,
The pdf links don’t seem to work for me, but I found the plots on the linked thread.
Even if the scan works with the alternative minimizer, it’s probably still worth understanding what goes wrong in the default Minuit fit, since it suggests some odd behaviour in the NLL function.
That’s not to say the GSLMultiMin scan is wrong, but we don’t have as much experience with it. The fact that it also changed the +1sigma uncertainty is odd.
So to understand the Minuit scan it would be good to check a couple of things:

Do you see the same jump if all constrained systematics are frozen to their best fit value?
If no, would be good to study how the NP values behave in the scan in the region around the bump. You can add --saveSpecifiedNuis all to get the profiled values in the ouput, and then see if any of them jump or have odd behaviour in this region.

cheers,
Andrew

Hi Rob,

Robert Stephen White:

(1) we tried freezing the systematics that resulted in the largest impacts/pulls, and this had no effect or the bump actually got larger in the plot.

There’s no reason a-priori why this has to be caused by the systematics with the large impacts. Probably best to check all of them.
The scans of the NP values vs mu in the slides is exactly what I mean, but it would be good to check all the nuisance parameters for the problematic likelihood scan. I don’t know technically how you made these plots already, but I was suggesting to use --saveSpecifiedNuis as a way to get the needed values in one go as you perform the NLL scan for the POI with --algo grid. Then essentially from the limit TTree instead of plotting the deltaNLL vs mu, you just plot the nuisance parameter value vs mu. The hope is that one or more of the parameters show a clear jump in value around the point in mu where the NLL bump occurs.
