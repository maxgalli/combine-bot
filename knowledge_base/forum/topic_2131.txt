REPLY 4 (by amarini @ 2021-07-26T19:59:40.000Z):
*** Discussion title: Higgs Combination
Hi Jona,
for some reasons your bin counts (in data) are non integers (they should be).
inside the input file you gave me
TH1.Print Name  = data_obs, Entries= 1002, Total sum= 4.99999
fSumw[0]=0, x=-0.166667, error=0  (*underflow, ignored in combine*)
fSumw[1]=3.99954, x=0.175, error=1.9996
fSumw[2]=0.000468908, x=0.5845, error=2.16521e-05
fSumw[3]=0.999981, x=0.9095, error=0.999801
fSumw[4]=0, x=1.16667, error=0 (*overflow, ignored in combine*)
The TH1 kPoisson option of TH1 returns the value you see with these non integers counts.
(inside combine)
DEBUG GREP MEdata (0): 0.5: y=3.99954 bw=1 el=2.63224 eh=1.91865
DEBUG GREP MEdata (1): 1.5: y=0.000468908 bw=1 el=0 eh=1.84055
DEBUG GREP MEdata (2): 2.5: y=0.999981 bw=1 el=0 eh=0.841041
You can test it in plain ROOT as:
root [0] TH1D h ("name","name",10,0,10)
root [1] h.SetBinContent(1,0.999981)
root [2] h.SetBinErrorOption(TH1::kPoisson)
root [3] h.GetBinErrorLow(1)
(double) 0.0000000
root [4] h.GetBinErrorUp(1)
(double) 0.84104064
root [5] h.SetBinContent(2,1)
root [6] h.GetBinErrorLow(2)
(double) 0.82724622
root [7] h.GetBinErrorUp(2)
(double) 2.2995266
Apparently, the implementation in ROOT [*] uses a mix of the value in double and its int cast (= floor).  The int casting inside root makes the large difference for this because the it takes the values for the interval at 0 but computes the error as difference with respect to the nominal double value (except 0, whose lower error is hard coded to 0). I suggest to clean up the observation to be integer values. (which should describe better the likelihood, although I don't think this will make a large difference in the fit values)
Cheers,
-Andrea
[*] https://root.cern.ch/doc/master/TH1_8cxx_source.html#l08913
