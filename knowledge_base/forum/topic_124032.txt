REPLY 2 [ACCEPTED ANSWER] (by nckw @ 2025-04-23T10:11:17.931Z):
Hi Amitav,
I think this is related to whether or not the parameter is attached to one of the pdfs (shapes) you are using.
In the case of QCD_Rc_0x0_par0, I can see that there are many RooFormulaVars that are clients, picking one, QCD_Rc_0x0_LOW_bin_1-1, and following the tree up, I can see there are RooParametricHist2D objects that use it - so combine will be able to see that the parameter is indeed a parameter of the model
Now, instead if I try with QCD_Rc_1x1_par0, I can see again a set of RooFormulaVars that are clients, but trying to follow up one of them:
w->function("QCD_Rc_1x1_LOW_bin_1-1")->Print("V")

I can see the client list is empty. This means, that the model doesn’t contain that parameter.
A fix is to declare them in the datacard usign the extraArg syntax (see https://cms-analysis.github.io/HiggsAnalysis-CombinedLimit/latest/part2/settinguptheanalysis/#extra-arguments) as follows;
QCD_Rc_1x1_par0 extArg base.root:w
QCD_Rc_1x1_par1 extArg base.root:w
QCD_Rc_1x1_par2 extArg base.root:w
QCD_Rc_1x1_par3 extArg base.root:w

This will get rid of the warning, but don’t know if it will have a real effect since unless the parameters are doing something to the model, I don’t see how they are effecting the analysis. Perhaps there is a reason but you might want to double check that the mapping of these parameters to the pdfs is correct,
Best,
Nick
P.S, in text2workspace, if you include a verbosity flag (-v 3) you will find a .dot file produced. This can be interpreted with several different graph viewing tools, and that might help you debug the model better.
