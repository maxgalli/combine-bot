REPLY 2 (by ineuteli @ 2026-02-12T13:48:54.373Z):
Hi Alexis,
I think in Combine, the bin uncertainty of the input histograms for the expected model is only used to compute the MC bin-by-bin-uncertainties when you include autoMCstat.
The bin errors of the pre-fit histogram should be the full (symmetrized) pre-fit uncertainty, which includes the all systematic uncertainties, as well as bin-by-bin uncertainties computed by autoMCstat. These can indeed be large depending on your prefit uncertainties.
Typically the bin error of the post-fit histogram are typically much smaller because the uncertainties get constrained by fitting the data and the correlations are taken into account via the covariance matrix.
However, are you using autoMCstat? These issue reminds me something I have encountered in Combine review, and of a previous question:

fitDiagnostics blows up pre-fit error bars Statistical Interpretation Tools

--saveWithUncertainties \

In the output root file I find the TDirectory shapes_prefit. The histograms stored in there have significan…

Like in those previous cases, I noticed that the bin uncertainty of your input histogram is the square root of the bin content, sqrt(4e-9) = 6.32455532. This is probably an indication

you created/filled the histograms without setting TH1::Sumw2, or
you set the wrong bin error option, like kPoisson TH1::SetBinErrorOption. This sets the bin error to the (asymmetric) Poisson error, which is approximately square root of the bin content, sqrt(N). This should only used for real data to get the correct error bars in plots. I believe Combine does this for you for the data_obs output histograms.

For MC histograms, you should always set the bin error option explicitly to TH1::Sumw2,
which keeps track of the sum of weights squared, and sets the bin error to the square root of the sum of weights squared.
The problem is that this bin error of the input histogram is wrong and too large, leading autoMCstat to create wrong and huge bin-by-bin uncertainties. This is because it computes the thresholds and size of the statistical uncertainty from the histogram bin errors with the Barlow–Beeston-lite algorithm, as you can read in detail in the Combine manual.
When I saw this in Combine review the issue arose from histograms created with Coffea. Whatever framework you use, my suggestion is to use TH1::Sumw2 before filling the MC histograms.
Cheers,
Izaak
P.S.
As this has happened several times, we should probably add a check of the bin error options and if TH1::Sumw2() was set  in the autoMCstat algorithm to warn users.
P.P.S.
I actually think that TH1::Sumw2 be the default in ROOT upon creation, depending on the default global settings or fgDefaultSumw2, or when filling.
>>> from ROOT import TH1F
>>> h = TH1F('h','h',10,0,10)
>>> for i in range(100): x=h.Fill(0.5,0.01)
>>> h.GetBinContent(1), h.GetBinError(1)
(0.9999993443489075, 0.09999999999999998)

REPLY 9 (by ineuteli @ 2026-02-13T13:03:55.004Z):
Hi Alexis,
As you said the fit converges, I am not sure what is happening there without looking into your datacards and shapes. The bin errors for the input look a lot more reasonable, i.e., the bin errors are smaller than the bin content, indicating you have good enough MC statistics.

Alexis Kalogeropoulos:

How is it even possible that the BinContent changed just by removing the autoMCStats? Any ideas?

As said above, it is generally expected that the pre-fit uncertainty changes in each bin. This new uncertainty is the total (symmetrized) pre-fit uncertainty, which are basically all systematic uncertainties summed quadratically, without taking into account any correlations. These uncertainties include the bin-by-bin uncertainties as well if you included autoMCStats, because these are added as nuisance parameters.
In your case something went awry, probably in the fit, but there’s no way of telling without the data cards and inputs…
You could try several things yourself:

Run ValidateDatacards.py to check for obvious mistakes in the shapes and their variations, see https://cms-analysis.github.io/HiggsAnalysis-CombinedLimit/part3/validation/.
Removing one by one systematics until pre-fit uncertainty looks reasonable and then try to understand what went wrong. You can simply remove the nuisance line from the datacard or use --freezeParameters.

Perhaps someone else has an idea?
Cheers,
Izaak
