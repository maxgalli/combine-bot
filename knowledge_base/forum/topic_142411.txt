REPLY 5 (by nckw @ 2026-03-01T14:20:06.012Z):
Thanks Giacomo,
I am no expert in theta and haven’t used the MarkovChainMC tool in some time, however, I have tried playing with the simple model (just the one with a simple scaling parameter sigma_t_ch and I noticed that the acceptance of the chains is rather low with the default settings (only a few %). This can be improved usually be reducing the step size in the proposals, which in combine can be achieved with the option --propHelperWidthRangeDivisor = X, when using the default proposal algorithm.
By setting X to some value > 1 (I found fairly good acceptance with 25), the size of the Gaussian proposal is reduced by that factor.
Note that when plotting the results of the chains (using the --saveChains option), the burn-in options have no effect so one needs to apply this by ignoring the first chunk of the chains. There’s an example on the combine docs as to how to pull out the results and plot them. Its also useful to plot the chains vs iteration to make sure they stabilise at some point. The example attached is what I get for the parameter sigma_t_ch if using the following options : --tries 200 –saveChain -i 30000 –propHelperWidthRangeDivisor=25. The left histogram is the results after throwing out the first 50% of the chains (although thats probably a bit high).
I would recommend playing with this option and making sure (with adding verbosity in the combine command) that the acceptance rate of the chains is more like 10-20% rather than only a few %.
Let me know if it helps at all,

REPLY 7 (by nckw @ 2026-03-04T16:01:05.810Z):
Hi Emil,
Thanks for trying the options - indeed, it looks like things have improved somewhat. In answer to your questions

We typically advise using more chains than increasing the length of the chains, but that really assumes that things settle fairly quickly. I am not sure why in your case the chains aren’t converging. I tried just a simple fit (-M FitDiagnostics) and this seems to complain about not being able to calculate errors. You have uncertainties called “cerr1/2” in your model, which I notice if I freeze them, the fit becomes stable. Do you have more details about these? I noticed than in some cases the shapes are “one-sided” in that both up/down variations are sometimes on the same side compared to the nominal.
I would usually recommend trying some more simple algorithms to make sure everything looks ok (eg some simple likelihood scans/fits) which you can try. But in this case, I haven’t found anything super suspicious yet - aside from the cerr1/2 nuisances above. I took this opportunity to add a chains debugging tool for combine (you can find details here :  https://cms-analysis.github.io/HiggsAnalysis-CombinedLimit/latest/part3/commonstatsmethods/#computing-the-observed-bayesian-limit-for-arbitrary-models, scroll down to the “debugChains.py" script.) which you can use to produce traces of the chains and see if they are converging after some point or not. You might find that useful.

I will keep investigating. One thing to note is that I am not sure your theta results make sense either since in many cases, the nuisance parameter posterior is centered around 1, instead of 0 (also in combine). At least in combine, the way that lnN and shape nuisances are implemented, I would expect them to be centered around 0, unless significantly pulled by the data.
All the best,

Yes that is the default option. The proposal is one that has a very high acceptance (relative to others) rate for the chains. I must admit, i’m not an expert really on this method, but it seems to be “common wisdom” that this method gives fairly stable results. However, clearly in this case its not working. For something closer to what Theta is doing, you can try the “fit” proposal, which performs a fit and then uses the resulting covariance matrix to create a multivariate gaussian as the proposal. If you use this option, i’d probably include --updateProposalParams=1 so that the mean moves around with the accepted point. The acceptance will be less than the default but I would not expect it to be zero. One additional thing that may help is to restrict the ranges of the nuisance parameters (by default the log-normals have a range of $\pm$7 in combine!). You can try changing the range of some parameters by hand or using the --cropNSigmas option.

Full FitDiagnostics shows that while all the pulls are close at 0, some nuisances (Barlow-Beeston especailly), significantly constrain the fit

This could be a guide to which parameters you might want to change the range of - if they are strongly constrained.
