Usually it's not a large number of parameters per-se that causes this - it only takes one parameter with an oddly shaped or discontinuous NLL curve, or two highly (anti-)correlated parameters to trigger this. Though it can be with a large number of parameters this is more likely to happen.
There are a couple of things you could try and look at:
- plot the unprofiled NLL vs each parameter: https://cms-analysis.github.io/HiggsAnalysis-CombinedLimit/part3/debugging.html#analyzing-the-nll-shape-in-each-parameter and look for anything odd
- Run with -v 3 and see if Minuit gives any hints in the verbose output from the fit

REPLY 3 (by ncsmith @ 2021-07-27T20:09:29.000Z):
Another possibility is parameters with a wide order of magnitude of uncertainties. It’s a bit of a catch-22 though because you can’t find the uncertainties until you invert the hessian.
Assuming most mass points in the grid have similar behavior, what I would suggest is to compare the ratio of maximum and minimum uncertainty on any free parameter in the fit result for one of the successful fits (look at fit_s in fitDiagnostics.root), ratios larger than ~1e3 start to be problematic. The easiest way to fix it is to rescale the offending parameters.
~Nick

From these observation I would conclude that the issue is caused by the large number of free parameters to be fitted.

REPLY 6 (by ncsmith @ 2021-08-10T17:34:33.000Z):
Hi,
I don’t see any immediate conceptual issue with restricting which bins you fit as a function of mass hypothesis. It is no different than having two analyses setting limits on two different models using different final states. It would be necessary to report the mechanism by which you trim bins then, as it is in effect part of the event selection.
That said, it is a bit unfortunate to have to employ this procedure to work around what appears to be a technical issue with the model and/or the minimization. If you like, I would be interested in taking a look at the model (cards+inputs) to see if it really is just a numeric issue.
~Nick
