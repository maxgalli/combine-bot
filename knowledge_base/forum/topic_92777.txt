REPLY 2 (by pkeicher @ 2025-01-07T16:26:14.252Z):
Hi @trmishra ,
I’ll have to look into this a little more closely, but I noticed that you run your blinded AsymptoticLimits in [0] with the -t -1 option. Note that this is not recommended for this method, see e.g. the docs under blinded limits.
Can you try running your command with --run blind instead of -t -1? Just to verify that it’s not a unwanted effect from this set of options.
Additionally, can you run the text2workspace.py script on your combined datacard and see if that runs correctly? This is also done under the hood if you give combine the datacard itself, but you might be able to gain further insights about a potential problem if you can’t build the workspace in the first place.
Side note: Particularly for larger combinations (such as full Run2 combinations), it’s generally recommended to run the text2workspace step once in a separate command. That way, you don’t have to spend the time it takes to create the workspace for every method you run with combine.
Hope this helps! Cheers,
Philip

REPLY 4 (by pkeicher @ 2025-01-08T09:57:01.441Z):
Hi Tribeni,
thanks for the follow up. I had a closer look at your full Run2 card, and noticed that the signal process SUSY is empty in many channels. I assume that’s because most of your channels are actually control regions, right?
I did a very quick test where I removed the gmN uncertainties to simply for the model and also remove the empty SUSY processes and their associated stat uncertainties. I was able to successfully run your AsymptoticLimits command afterwards. Based on this, I think the problem might be that you try to evaluate lnN uncertainties on empty processes. Can you maybe remove the empty processes at your end and try again (also including the gmN uncertainties where appropriate of course)? Of course, if a given uncertainty only has an effect on empty processes, I’d also recommend removing it for this test.
