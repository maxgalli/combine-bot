In our analysis, we combine datacards together with combineCards and
text2workspace before running MultiDimFit with the signal region masked
to perform a fit between control region data and MC. We then unmask the
signal region in the workspace before running limits on the resulting
snapshot. However we have recently noticed some discrepancies on the
resulting limits depending on the Combine version.
So in one category we had a limit of 1.4844 (50% exp) using a clone of
the Combine repo from March this year (branch 102x, on CMSSW_10_2_13). I
then tried updating my Combine area using the recommended tag [1] but
have consistently had issues with the MultiDimFit, resulting in a limit
of 0.0928 for this particular category (and all other limits in the
analysis being far smaller than they used to be). I have struggled to
find what is causing this issue, and the commands we use do not suggest
the fit fails necessarily in these steps. Trying to checkout the tag at
different stages of it's commit history hasn't helped either.
So then I returned to branch 102x, pulled the latest version of Combine
[2], and several of our categories have clearly different limits to
before the pull, in some cases increasing significantly (as in, for the
category we previously had limit 1.4844 we now get 1.4141. Although this
is one category, in our supercategories we're finding the limits
increase by ~few percent too).
Firstly, between the two versions of Combine branch 102x, could we
expect that the limits can change a lot before and after a git pull? I
would expect maybe a small change but not ~5% in overall year limits.
Secondly, is there a known issue with the tag recommended in the Combine
docs?

combine higgsCombine_MLECRData.MultiDimFit.mH125.root --snapshotName MultiDimFit --run expected --bypassFrequentistFit -m 125 --setParameters "${mask_SR_channels_null}"

Hi Rob,
It's not 100% clear to me how many different limits you actually end up
with, but I'm assuming there are 2 different limits for the same
category (one with the old 102x branch at an unspecified time in March,
one with more recent versions of the 102x branch).
One thing we changed just before releasing the v8.2.0 combine tag is
making the analytic minimisation for parameters generated by autoMCStats
the default, but this can lead to changes in the fit results when using
the channel masking. Assuming that you are using autoMCStats, can you
try to add --X-rtd MINIMIZER_no_analytic to the MultiDimFit step and see
if, using the snapshot you create from that, the "old" limits are
recovered?

REPLY 4 (by adewit @ 2021-08-11T09:42:55.000Z):

Hi Rob,
Okay, glad this solves the issue! --X-rtd MINIMIZER_no_analytic turns
off analytic minimization of the parameters created by autoMCStats. This
was default up to combine v8.2.0, but as analytically minimizing these
parameters is much faster than handing them over to minuit, and this
method was extensively validated, we made analytic minimization the
default.
For most analyses the change was transparent, however for channels that
are masked from the likelihood the analytic minimization of the
autoMCStat parameters still takes place, whereas these parameters would
not have been moved from their initial values in the case where
everything was handled by minuit. In this case it means you had two
slightly different snapshots to start the limit calculation from.
This effect is only present when channels are masked, so it should not
have an impact on any final results.
Best,
Adinda
