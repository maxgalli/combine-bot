QUESTION (by woodson @ 2022-01-22T13:53:35.976Z):
Hi all,
In the HH combination for the VBF signal interpolation to different (kl, CV, C2V) values, we use a linear transformation applied to 6 templates in the default model. See https://indico.cern.ch/event/904966/#10-di-higgs-physics-model and https://gitlab.cern.ch/hh/tools/inference/-/blob/master/dhi/models/hh_model.py for more information. The full formula looks something like this:
qqHH_CV_1_C2V_0_kl_1*(0.5*C2V**2 + 0.833*C2V*CV**2 - 2.33*C2V*CV*kl - 1.33*CV**4 + 2.33*CV**3*kl ) +
qqHH_CV_1_C2V_1_kl_0*(1.2*C2V*CV**2 - 1.2*C2V*CV*kl - 0.2*CV**4 - 0.3*CV**3*kl + 0.5*CV**2*kl**2) +
qqHH_CV_1_C2V_1_kl_1*(-1*C2V**2 + 5.133*C2V*CV**2 - 3.133*C2V*CV*kl - 4.133*CV**4 + 5.133*CV**3*kl - 1*CV**2*kl**2) +
qqHH_CV_1_C2V_1_kl_2*(-0.6*C2V*CV**2 + 0.6*C2V*CV*kl + 0.6*CV**4 - 1.1*CV**3*kl + 0.45*CV**2*kl**2) +
qqHH_CV_1_C2V_2_kl_1*(0.5*C2V**2 - 1.166*C2V*CV**2 + 0.66*C2V*CV*kl + 0.66*CV**4 - 0.66*CV**3*kl ) +
qqHH_CV_1p5_C2V_1_kl_1*(-1.066*C2V*CV**2 + 1.066*C2V*CV*kl + 1.066*CV**4 - 1.066*CV**3*kl )

For some values especially in the 2D (C2V, kl) at CV=1, the coefficients can become very large (and negative). For example for C2V=0.5, kl=15, the formula evaluates to:
16.7*qqHH_CV_1_C2V_0_kl_1 +
99.4*qqHH_CV_1_C2V_1_kl_0 -
173.3*qqHH_CV_1_C2V_1_kl_1 +
100.8*qqHH_CV_1_C2V_1_kl_2 -
4.8*qqHH_CV_1_C2V_2_kl_1 -
7.5*qqHH_CV_1p5_C2V_1_kl_1

Due to limited MC stats and the large coefficients involved, we sometimes see the resulting summed signal Pdf goes negative in some bins and, if the background Pdf is close to zero, the total Pdf goes negative and the fit fails.
We’re wondering how to solve this problem in the general case as several analysis channels may experience this. One thought is to add a max(0,  summed_signal_pdf) “protection” perhaps as a wrapper C++ class, but we’re open to any suggestions from experts.
Let us know what you think and thanks!
Javier

REPLY 2 (by amarini @ 2022-01-22T15:07:58.677Z):
Hi Javier,
in general, the problem with the total entries in a certain bin becoming negative, lies in the fact that the likelihood function is not defined.
(I.e., a probability density function or Poisson counts are defined only in the region where the total number of expected events are positive).
If you have an observation of 1 event in such bins, the likelihood function will assume larger and larger values the closer you approach a total number of expected events closer and closer to 0. [*]
So ditto, there are two cases:
The first case is when the coefficients are known with fairly good precision, and in such case you can state that the model is not valid in that part of the parameter space.
If the coefficients are not well known, you can add such protection (maybe also a 10^-x should be safer, depending on how the datacards and those bins are expected to be),
but it is very likely that you are in a part of the phase space where the model cannot give an accurate prediction anymore.
It would be useful to understand how important is that ph. space region / bin to the overall result (this can be done by removing / masking those bins and assessing if the results changes significantly).
Ideally, it would be better to generate some MC events around those region of phase space, to bound the correct predicted events.
Notice that for what I said before [*], it is also unlikely (but I would have to think about it a bit more) that a constrained uncertainty on those numbers solves the “problem”, since the likelihood function in that portion has no natural constrain and pushes the postfit events to 0, without a natural minimum.
Cheers,
-Andrea
P.S. It is also preferred to call it as max(0. [dot], … ) , such that c++ always resolved it as <double, double> instead of <int,double>. I saw some versions of c++(/root/roofit) where this may cause problems, compilation errors, even in TFormulas.

REPLY 3 (by ncsmith @ 2022-02-07T18:58:21.246Z):
Hi,
Javier and I worked a bit offline on this subject and I came to the conclusion that the method of [1] has some caveats that aren’t considered: the fact that the per-basis-point templates are stochastic can mean that the result is negative for some values of the parameters. A more careful treatment can avoid this, and I’ve described as much in the attached note that I’ve been putting together.
I’ve seen this same treatment elsewhere, as well as an alternative approach that adds up the per-event quadratic expansion of a single reference point in coefficient space rather than making an interpolation from basis points, as used in e.g. TOP-19-001. Both are described in the note.
note.pdf (288.3 KB)
I would be very interested to hear others’ solutions to the problem.
