REPLY 2 (by agilbert @ 2022-06-23T07:35:03.056Z):
Hi Jyoti,
Unfortunately the autoMCStats feature currently only works when all input shapes are TH1 based. It looks like you have a custom RooFit pdf in your model (RooDijetBinPdf).
There are a few technical reasons why applying this on arbitrary pdfs is challenging - not least because AFAIK RooFit has no mechanism to associate a set of per-bin uncertainty values to arbitrary binned pdfs, so we wouldn’t know what uncertainties to model. With TH1 it’s straightforward - we can extract the bin error.
In the future, we might consider whether we can support more general cases. What sort of input do you have for the signal template? RooDataHist, RooHistPdf, or something else?
For now, you will probably need to introduce standard shape uncertainties on the signal to model the uncertainty for each bin.
cheers,
Andrew

REPLY 4 (by agilbert @ 2022-06-23T10:16:47.192Z):
Hi Jyoti,
What you can do is introduce a “shape” uncertainty for each bin of the signal distribution, following a similar prescription to https://cms-analysis.github.io/HiggsAnalysis-CombinedLimit/part2/settinguptheanalysis/#binned-shape-analysis
Except here you would provide alternative RooDataHists for the Up and Down variations - in each one changing the yield of a single bin by its stat uncertainty. You already have appropriate argument in the datacard for this to get picked up, you just have to make sure the naming convention matches (wExcitedQuarks2016:$PROCESS_$SYSTEMATIC).
cheers,
Andrew

REPLY 6 (by agilbert @ 2022-06-27T11:40:23.616Z):
Hi Javier,
It seems that RooDataHist can now store sumw2 values: https://root.cern.ch/doc/master/classRooDataHist.html#a3303cccda866fdc0e5953989a069f480, so this might make it even easier to parse the uncertainties.
The next challenge is then constructing the likelihood model. With TH1 inputs, each process is modelled as a CMSHistFunc object, and these are summed via the CMSHistErrorPropagator. It’s the latter that handles all the MC stat modelling, and deals with the analytic minimisation. The problem is right now this class can only handle CMSHistFunc input. Accepting arbitrary RooAbsReal/RooAbsPdf classes here will take some more work, with tricky side-effects (e.g. we would potentially lose some combine optimisations).
That said, we understand this is a desirable feature. I’m not sure if the best way forward is to hack something into the existing code, or take a step back and start from something more basic. There are some other potential improvements for binned analyses in combine that are in the pipeline - might be good to discuss all together, perhaps in an upcoming hcomb meeting.
cheers,
Andrew

REPLY 9 (by woodson @ 2022-07-01T18:24:27.000Z):
Hi Jyoti,
Yes, you have to give 1 up and down shape for each bin and MC process, so that you have an independent nuisance parameter for each independent statistical uncertainty. For example, if the nuisance parameter would be named mcstat{ProcessName}{BinNumber}, there should be two RooDataHists for that variation:
process_mcstat{ProcessName}{BinNumber}Up
process_mcstat{ProcessName}{BinNumber}Down
In which only that bin is varied up or down by the poison uncertainty.
Does that make sense?
Thanks,
Javier

REPLY 11 (by gortona @ 2022-07-05T05:59:55.240Z):
Hi @jbabbar,
No you will need to provide one up and one down RooDataHist for each bin of each template you want to apply this effect to. To make it clearer, if your ExcitedQuarks2016_Qstar histogram has 200 bins, in principle you need to provide 200 up and 200 down RooDataHists.
The idea is that since this effect is independent for each bin (meant as actual bin of an histogram here, not as category) you need an independent shape variation for each. If you have just one up and one down as in the screenshot you attached you would be varying all of them simultaneously.
Of course it is best to reduce this number by not applying this uncertainty to those bins that are enough populated.
On this point, may I ask you where this statistical uncertainty is coming from? Usually signal MC is well populated and there should be no need to apply this effect to signal templates.
Best regards,
Giacomo
