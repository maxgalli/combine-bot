REPLY 4 (by kcormier @ 2023-04-24T13:45:34.891Z):
Hi Hugo,
Thanks for the additional info, this is more clear to me now. Sorry, I had initially misinterpreted what you meant; I thought you wanted a single bin to be controlled by a rate parameter which had no correlations to other bins/channels.
But now, I think I understand that you are trying to derive a per-bin correction factor which you want to estimate from a set of control regions. So the idea is that based on how the processes scale in these control regions it will also scale the process(es) in your signal region(s). But in principle this scaling is kinematic dependent, so you want it to be derived per-bin, rather than simply per-process. Please correct me if that’s wrong.
As I said, there is no way to specify in the datacard directly that a rate parameter affects only one bin of a distribution. However, there are a number of ways to achieve it. One way which is documented with an example taking you through the steps, including adding systematics etc, is using a RooParametricHist to implement this, you can see the example here [1]. When you’re using the RooParametricHist you are able to define parameters which scale individual bins, and then this can be saved in a workspace and passed to the combine datacard.
Another option is to artificially define “processes” which only contribute to individual bins, e.g. Vjets_pt_30to50, Vjets_pt_50to70, etc… the histograms defining these “processes” would then only have entries in the corresponding bin, and be empty in all others, but together they would include the whole Vjets spectrum, for example. Then you can easily assign rate parameters to them in the datacard as you please.   But using the first method defined above may scale better and help keep the datacard more readable.
I think that covers your use case, but let us know if something is missing or unclear.
Cheers,
Kyle
[1] http://cms-analysis.github.io/HiggsAnalysis-CombinedLimit/part3/nonstandard/#rooparametrichist-gamman-for-shapes

REPLY 7 (by kcormier @ 2023-05-08T14:05:11.363Z):
Dear Hugo,
What this is telling you is that in performing the fit, the parameter values are coming up against boundaries of the parameter range, which can be a problem.
This can happen for many reasons,  it can be as simple as a poor choice of parameter ranges, but there are many potential reasons.
Taking a quick look at your first datacard (your [3], vjj.txt),  I see two potential issues.
If I’ve understood correctly, you have two separate channels (gamma_2018_SR, and Z_2018_SR), each with 5 bins. Then you have a total of 15 freely floating rate parameters, one for each bin in each of the channels, and for each of the 5 bins another which links the processes in the two regions.
This can cause problems because your system is underdetermined. If you have only 10 data points (5 + 5 bins) but 15 free parameters, then there are many degenerate solutions, and the mimimum of your function will span an area of the parameter space, which would explain the boundary warnings.
My understanding of what you wanted to do is to have a single rate parameter which simultaneously controls the scaling of all process. If you also want another nuisance parameter controlling the individual process cross sections, you will need at least one of those two parameters to be constrained somehow. Otherwise there is a degeneracy in the function which cannot be resolved without some other constraint.
Another potential issue I see is that the limits on the rate parameters you have set can be quite small (in some cases as small as a 3% change). Even if there is no issue with the degeneracy, if the best fit point does not constrain these values well, or lies outside of the chosen range, this will cause problems. So you may need to adjust the choice of range, depending on the model and the data.
I haven’t had a chance to look at your other implementation yet to see if might have the same issues. But maybe this could be the source of your difficulty?
Cheers,
Kyle

REPLY 10 (by kcormier @ 2023-05-12T13:02:54.318Z):
Dear Hugo,
Sorry for the slow response.
I think there is still a degeneracy in this setup. If n_freefloating > n_Data, then things are definitely a problem. but even when n_freefloating < n_Data, this doesn’t guarantee that there is no degeneracy.
it depends on what the actual directions the free floating parameters pick out are.
Here, I think you have a problem because there is still a degeneracy, in particular, the directions
rate_bin1  + rate_bin2 + rate_bin3 + rate_bin4 + rate_bin5

xsec_g1jet + xsec_dy_nlo

are the same.
So there is no way for this to be resolved by the fit, one can always compensate for the other. Even though n_params < n_data, there is a degeneracy.
If one of the signal parameters happens to lie in the space spanned by the rateparams that will also cause issues. Or if one lies outside of the space spanned by the rateparams, but is mostly inside that space it will still be highly correlated and could cause issues.
I think your particular signals may be fine in this case, but just something to note in case you still run into problems after fixing the issue with the degeneracy.
A few general points, related to this that may be of help.

It’s good to check FitDiagnostics or using MultiDimFit with the -grid option to scan some parameters to see how the fit is behaving when things are failing.

From FitDiagnostics you can see things like if the fit is finding a positive definite covariance matrix, if not is is indicating a problem. (In this case it doesnt find a positive definite covariance matrix for example, and has to force it to be so)
The scan can show you the shape of the minimum as a function of a parameter and you often see some pathologies. In this case if you scan the NLL as a function of, say, one of the rate parameters it will give very strange results, such as a mostly flat minimum due to the degeneracy. (Sometimes you can see it increase, due to e.g. limits on some parameters breaking the degeneracy). But if you fix one of the parameters to break the degeneracy you’ll start seeing a healthier NLL scan again.

This is my personal opinion, so someone can jump in if I’m missing something. But in general, I think with unconstrained parameters you should question whether its truly nuconstrained or not. I find that that this is often a poor representation of what we want to be done, which may be something like implying a wide range of compatible values (where I’d suggest using a wide constraint). But often the combination of flat constraint + hitting boundaries is telling you that there is an issue. In my experience, hard cut-off constraints usually don’t represent the actual situation well, and can cause numerical issues.

I hope that helps,
Kyle
