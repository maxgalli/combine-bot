QUESTION (by nivanden @ 2026-01-08T11:16:55.497Z):
Dear Combine experts,
For TOP-24-008 we are extracting 68% and 95% limits with toys as we have a quadratic and higher order dependence on the POIs. It is important that we can distinguish between a local minimum and global minimum between positive and negative values of the POI so we are looking closely at these toys and refitting the toy datasets. Here I noticed some behaviour I could not quite understand.
I run the toy generation and likelihood evaluation as follows:
combineTool.py -M MultiDimFit workspace.root --algo fixed -t 5 -s 1017 --toysFrequentist -P at --redefineSignalPOIs at --setParameters at=-1,bt=0  --fixedPointPOIs at=-1,bt=0 --freezeParameters bt --cminDefaultMinimizerStrategy 0 -n toys_at_m1_bt_0 --saveToys

and this gives a best fit value for the toy with ID 1 (iToy == 1) at at=-0.983.
When I now rerun the fit (or perform a likelihood scan) of this toy 1, I notice this result is not the actual minimum. I rerun the fit as:
combineTool.py -M MultiDimFit workspace.root --algo fixed -s 1017 --toysFrequentist -P at --redefineSignalPOIs at --setParameters at=-1,bt=0  --fixedPointPOIs at=-1,bt=0 --freezeParameters bt --cminDefaultMinimizerStrategy 0 -n toys_at_m1_bt_0_REFIT -D "higgsCombinetoys_at_m2_bt_0.MultiDimFit.mH120.1017.root:toys/toy_1"

and I obtain a best fit value of at = -1.081. A scan of the likelihood also confirms the minimum lies at this -1.081. This behaviour is repeated for other toys. The problem with this, is that the test statistic value distribution that I want to extract from the toys might also be affected (I only tested individual toys at this point), as the value of the test statistic at my fixed point differs between both fits.
As a last test, I ran over the complete toys file:
combineTool.py -M MultiDimFit workspace.root --algo fixed -s 1017  -P at --redefineSignalPOIs at --setParameters at=-1,bt=0  --fixedPointPOIs at=-1,bt=0 --freezeParameters bt         --cminDefaultMinimizerStrategy 0 -n toys_at_m1_bt_0_Refit_AllToys --toysFile=higgsCombinetoys_at_m1_bt_0.MultiDimFit.mH120.1017.root -t 5 --toysFrequentist

And this reproduced most of the results of the first command (but not all).
My expectation is that all three fits should give the same result. Why is this not the case and which result is actually correct?
Kind regards,
Niels

REPLY 2 (by gallim @ 2026-01-27T13:56:19.135Z):
Hi Niels,
here are my two cents (but I would be happy if someone else could confirm and/or correct this). When you upload the toy with ID 1 using -D, combine doesn’t seem to also upload the corresponding snapshot that was created and saved in the previous step (i.e. toys/toy_1_snapshot).
However, this should correctly happen when you run with --toysFrequentist (see here). Based on this, I would say that the correct way to go would be to save the toys and then use them with the --toysFrequentist option.
What I’m still not sure is why only most of the results are reproduced this way, but not all (as you wrote in the thread).
