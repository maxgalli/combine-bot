Regarding the GoF, I suppose this is the same or similar to the rhalphabet method we used in ggH(bb)/(cc), right? I think the idea to use GoF was to have a statistic that should (eventually) follow a chi-square distribution. I guess since in the end we use toys (because we don’t want to have to rely on our determination of the degrees of freedom) we don’t really need the two GoF values to obey chi-square and could just take the raw nll values out of each fit to build the distribution and compute the quantile of the observed value. Though it won’t be an F distribution.
As an aside, and perhaps you already implemented it, but something that might affect fit stability in this method that we saw in the past was that it is necessary to rescale the fail bin yield free parameter to have approximately unit variance.

REPLY 8 (by agilbert @ 2022-04-06T10:10:23.884Z):
Hi,
I think I understand the issue.
First, on the difference between FitDiagnostics (or MultiDimFit) and GoF. I remembered there is a small difference - by default FitDiagnostics has some optimisation for the constrained parameter ranges, which is not available in the GoF routine. You can disable it with --X-rtd OPTIMIZE_BOUNDS=0, then it will also fail in FitDianostics.
Now I don’t think this range optimisation is actually that relevant. Your minimisation fails because Minuit exceeds the maximum number of calls it is allowed to make, and I think this happens because the FCN values are rather large in magnitude, and so the edm numerical precision is lower than normal. Best way to deal with this might be to have combine run a loose initial fit before the main fit, by adding --cminApproxPreFitTolerance 1000 or --cminApproxPreFitTolerance 100.
cheers,
Andrew
