REPLY 7 (by stclark @ 2022-09-05T12:49:21.903Z):
Thanks for all the help. I tried all the suggestions here and even tried everything in CMSSW_10_2_1 to no avail, but no it looks like I have found the (somewhat silly) solution:
I have been using a script that reads a config file and writes the datacard. I modified a config file from a previous analysis that was producing functioning datacards and I had one that was producing these broken datacards shared here. I found that the only difference between the two config files was a line like this:
from bad config file:
'RooMultiPdf::DIPHOM_multi(pdf_index,DIPHOM_bkgdijet, DIPHOM_bkgatlas, DIPHOM_bkgmoddijet)',
from good config file:
'RooMultiPdf::DIPHOM_multi(pdf_index,DIPHOM_bkgdijet,DIPHOM_bkgatlas,DIPHOM_bkgmoddijet)',
The only difference is the spaces after the commas. This was then being split on the comma and then stored in a list, In the bad case, the list stored variables like " DIPHOM_bkgatlas" with a leading space. This caused the output .txt datacard to look identical to one without the leading spaces, but clearly broke something in the corresponding .root file.
I don’t know exactly what is going wrong in the .root file, but it appears as if the problem has been solved. Fortunately it’s an easy solution though it caused much headache
