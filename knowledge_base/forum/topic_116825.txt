I think the interpretation of the p-value is the answer here. You calculate these p-values from the distribution (or pdf) of your test statistics q based on a-posteriori pseudo data that is generated by sampling your model parameters from their post-fit uncertainties. Since you try to minimize q, large values indicate a stronger tension between your model and the (pseudo) data. The p-value can be interpreted as the likelihood to find a value of q that is as large or larger than the value that you test (typically the value of the test statistics you obtain with the observed data q_{obs}). Usually, we are concerned about large values of q which would indicate that your model cannot describe the data very well.
Due to the statistical nature of our measurements, there is of course always a probability that you obtain an observed dataset that you can’t describe well, even if it would work for other data. That’s why we first sample the pdf of q to formulate an expection, and then derive the probability that I mention above. The threshold until which you accept the hypothesis “my model can describe the data sufficiently” is somewhat arbitrary and should be set before the actual test. In practice, many analyses use a p-value of 5%, which corresponds to a deviation of q_{obs} to the bulk of the expected values of q of 2 \sigma.
Generally speaking, this means that a large p-value indicates good compatibility of the model you test with the observed data, given the expected fluctuations within the model. One of the core prerequisites of this though is that q_{obs} isn’t too different from the values in your pdf. For example, if q_{obs} = 1 but all the values of your pdf are significantly larger, this would indicate that the model can describe the observed data set much better than pseudo data you obtain from sampling from the model parameter uncertainties. This can happen if these uncertainties (and thus the fluctuations) can become very large, which isn’t a problem per se but could indicate some instabilities in the minimization or some large tension in your statistical model.
I ran your datacard with much more pseudo data (you can actually parallelize this quite nicely with the combineTool.py and some compute cluster like htcondor with e.g. this (*)), and it seems to me that your value of q_{obs} is still part of the overall pdf (even though it’s in a region with very low population). I think this is ok - of course, if you want to try different binnings that bring q_{obs} and your pdf of q closer together that’s also fine, but I think this statistical test doesn’t necessitate that as it is.
(*)
Command to generate toys distribution in M batches of N toys
for i in {0...$M}; do
combineTool.py -M GoodnessOfFit -s -1 [your other options] -t $N -n toys_${i} --job-mode condor
done

Note that -s -1 will use random numbers as random seeds for the toys to ensure statistical independence of the toys across batches.
